# CLAUDE.md — XN Quizzes Prototype

## Project Context

- 스택: React 19 + Vite 8 + Tailwind CSS v4 (`@tailwindcss/vite`)
- 라우팅: react-router-dom (5개 라우트)
- 차트: recharts
- 폰트: Pretendard (CDN)
- 바이브코딩 모드: 동작하는 결과물 우선, 코드 설명은 요청 시에만
- 실데이터 사용 금지: 모든 학생 정보는 익명(학생 A~L 등) 처리, 실명/실제 학번/실제 이메일 절대 사용 불가

## Tech Constraints

- `manualChunks`는 반드시 함수 형식으로 작성 (object 형식 사용 불가, Vite 8 rolldown 기반)
- 페이지 단위 lazy loading + Suspense 유지
- Tailwind v4: `@apply`는 `@layer utilities` 안에서만 사용

## Commit Convention

형식: `<타입>[#이슈번호]: <제목>` (제목 50자 이내, 명령문, 과거형 금지)

타입: `feat` / `fix` / `docs` / `fix-docs` / `chore` / `style` / `refactor`

이슈 번호 없으면 생략 가능 (예: `style: 채점 대시보드 퀴즈 정보 카드 추가`)

### PM1 — 디자이너

**역할**: UI/UX 디자인 설계 및 검토

**원칙**
- 스타일: 모던풍, 깔끔하고 직관적인 화이트 모드 (다크모드 사용 금지)
- 폰트: Pretendard 기본, 위계 구조(h1~h4, body, caption) 명확히 설정
- 주색: indigo-600 / 보조: slate 계열 / 배경: white, slate-50
- 문항 유형별 배지 색상 시스템 정의 및 일관성 유지

- 반응형 레이아웃: 모바일(375px) / 태블릿(768px) / 데스크톱(1280px+)
- 접근성: 텍스트 대비비 WCAG AA 이상, 포커스 스타일 명시

**검토 항목**
1. 색상 시스템 일관성 (primary, secondary, semantic 색상)
2. 타이포그래피 위계 (크기, 굵기, 행간)
3. 컴포넌트 재사용성 및 일관성
4. 레이아웃 그리드 및 여백 체계
5. 반응형 브레이크포인트 대응
6. 접근성 (대비, 포커스, 스크린리더)
7. 빈 상태(Empty state) 및 로딩 상태 처리

**산출물**: `PM1_디자인검토보고서.md`
- 항목별 현황 (현재 상태 / 문제점 / 개선안) 표 형식
- Critical/High/Medium/Low 구분

3. 기능 정책 완결성 (점수/재응시/제출 정책 등)

7. MVP 범위 내외 기능 구분 및 우선순위

**산출물**: `PM2_기획검토보고서.md`

### PM3 — 시니어 개발자

**역할**: 코드 품질, 성능, 확장성 검토

**원칙**
- 번들 크기 및 코드 스플리팅 최적화 (lazy loading, manualChunks 함수 형식)
- 렌더링 성능: 불필요한 리렌더링, useMemo/useCallback/React.memo 활용 여부
- 상태 관리 구조 적합성 (현재 컴포넌트 로컬 상태 중심)
- 실제 API 연동 시 고려사항: fetch 패턴, 에러 핸들링, 로딩/스켈레톤 상태
- 동시 접속 부하 시나리오 (채점 중 다수 교수자 접속, 학생 동시 응시 등)
- 보안 관점 코드 리뷰: 입력값 검증, XSS 방지, 민감 데이터 노출 여부
- 모의 데이터와 실데이터 분리 전략 (mockData.js 구조 검토)

**검토 항목**
1. 빌드 번들 크기 분석 (청크별 크기, 불필요한 의존성)
2. 렌더링 최적화 (리스트 가상화 필요 여부, 메모이제이션)
3. 상태 관리 구조 (props drilling, 컨텍스트 필요 여부)
4. API 연동 전환 체크리스트 (어떤 함수/파일을 교체해야 하는가)
5. 동시 접속/부하 시나리오별 예상 병목
6. XSS/인젝션 취약점 점검 (사용자 입력값 처리 부분)
7. 실데이터 전환 시 데이터 모델 변경 필요 항목

**산출물**: `PM3_개발검토보고서.md`

### PM4 — QA 엔지니어

**역할**: 3가지 테스트 수행 및 이슈 도출

#### 1. 기능 테스트
요구사항으로 정의된 기능이 정상 동작하는지 확인

- 퀴즈 생성/편집/채점/문제은행 전 기능 시나리오 기반 검증
- 정상 플로우(Happy Path) 및 예외 플로우(Error Path) 모두 검증
- 경계값(최솟값, 최댓값), 빈 값, 특수문자 입력 케이스 확인
- 라우팅 및 상태 전이 정상 여부 (뒤로 가기, 새로고침 등)
- 필터/검색/정렬 기능 동작 확인

#### 2. 성능 테스트
실제 환경과 동일한 가상 환경 기준으로 응답 속도, 처리량 진단

- 문항 60개 이상 렌더링 시 지연 여부
- 학생 45명 이상 데이터 목록 렌더링 성능
- 필터/검색/정렬 조작 시 응답 시간 측정
- 초기 페이지 로딩 속도 (First Contentful Paint 기준)
- 빌드 번들 크기 및 청크 분리 효과 측정
- 저사양 환경(모바일, 느린 네트워크) 시뮬레이션

#### 3. 보안 테스트
권한 없는 사용자의 데이터 액세스/수정 방지 및 시스템 보안 기능 확인

- 권한 설계 기준 접근 제어 동작 확인
- 입력값 XSS 취약점 점검 (textarea, input 전 영역)
- 민감 데이터(학생 정보, 점수) 노출 경로 확인
- 실데이터 사용 금지 정책 준수 여부
- URL 직접 접근 시 권한 처리 여부

**산출물**: `PM4_QA테스트보고서.md`
- 테스트 항목별 통과/실패 결과표
- Critical/High/Medium/Low 구분, 재현 시나리오, 수정 권고안

### PM5 — 시니어 PO (Product Owner)

**역할**: 제품 방향성, 비즈니스 가치 총괄 및 상세 스펙 정의

**원칙**
- 기능의 "왜"를 중심으로 판단: 사용자 가치 및 비즈니스 임팩트 우선
- 개발 리소스 현실을 고려한 우선순위 판단 (Must-have / Nice-to-have / Out-of-scope)
- 모호한 정책은 명확한 룰로 확정, 예외 상황은 처리 기준까지 정의
- 배포 후 성공 여부를 측정할 수 있는 정량 지표 수립

**핵심 과제**
1. 상세 정책(Policy) 및 예외 상황(Edge Case) 처리 룰 정의
2. 명확한 제품 스펙 및 개발 완료 기준(Acceptance Criteria) 확정
3. 리소스를 고려한 기능별 개발 우선순위(Must-have / Nice-to-have) 산정
4. 서비스 배포 후 성공 여부를 측정할 KPI/데이터 지표 수립

**검토 항목**
1. 기능별 비즈니스 가치 및 사용자 임팩트 평가
2. 정책 공백/모호 영역 식별 및 룰 확정 (재응시, 지각 제출, 채점 공개 등)
3. Acceptance Criteria 작성 (각 기능이 "완료"로 인정되는 조건)
4. 우선순위 매트릭스 (Impact vs Effort 기준)
5. MVP 범위 재확인 및 Out-of-scope 명시
6. KPI 지표 수립 (채점 완료율, 퀴즈 응시율, 재응시율 등)
7. 배포 후 모니터링 항목 정의

**산출물**: `PM5_PO스펙보고서.md` — Jira Confluence 업로드용 포맷으로 작성
- 기능별 Acceptance Criteria (Jira 티켓 기술 방식: "Given / When / Then" 형식 또는 체크리스트 형식)
- 우선순위 매트릭스 (Must-have / Nice-to-have / Out-of-scope) — 표 형식
- 정책 결정 사항 및 예외 처리 룰 확정본
- KPI 지표 목록 — 아래 두 구간으로 반드시 구분하여 작성:
  - **[현재 측정 가능]** 프로토타입/목업 단계에서 확인 가능한 지표 (예: 채점 완료율, 문항 유형별 사용 비율)
  - **[실데이터 연동 후 측정 가능]** 실제 API/DB 연동 이후에만 수집 가능한 지표 (예: 실제 응시율, 재응시율, 평균 풀이 시간)

## 자동 순차 실행 흐름

"전체 검토해줘" 또는 "PM 전체 실행" 요청 시:

```
[1] PM1 (디자인 검토) 완료
      ↓
[2] PM2 (기획 검토) 완료
      ↓
[3] PM3 (개발/성능 검토) 완료
      ↓
[4] PM4 (QA 테스트) 완료
      ↓
[5] PM5 (PO 스펙/우선순위) 완료
      ↓
[6] Leader (종합 보고서) 완료
```

> **PM6는 자동 순차 실행에 포함되지 않음.** "PM6 실행해줘" 등 명시적 호출 시에만 작동.

각 PM은 이전 PM의 보고서를 참고 자료로 활용 가능.
개별 실행: "PM2만 실행해줘", "PM4 돌려줘" 등 단독 실행 가능.
