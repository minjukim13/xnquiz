## 1. 핵심 사용자 시나리오 (Happy Path) 정합성

### 전체 플로우 진단

```
[정의된 흐름]
퀴즈 생성 → 편집 → 발행 → 응시 → 채점 → 결과 확인

[현재 구현 커버리지]
생성(O)  →  편집(O)  →  발행(△)  →  응시(X)  →  채점(O)  →  결과확인(O)
```

### 시나리오별 현황

| **시나리오** | **구현 여부** | **현재 상태** | **문제점** |
|---|---|---|---|
| S1. 교수자가 퀴즈를 신규 생성한다 | O | QuizCreate 구현 | 발행 후 실제 목록 반영 없음 (mock 고정) |
| S2. 교수자가 기존 퀴즈를 편집한다 | O | QuizEdit 구현 | 편집한 내용이 저장 후 목록에 반영 안 됨 |
| S3. 교수자가 퀴즈를 발행/게시한다 | 부분 | "저장 및 발행" 버튼 존재 | 발행 후 상태 변화(draft→open) 처리 미구현 |
| S4. 학습자가 퀴즈에 응시한다 | X | 미구현 | 응시 화면 자체가 없음, MVP 범위 밖으로 추정 |
| S5. 교수자가 채점한다 | O | GradingDashboard 구현 | 채점 저장 액션 없음 (저장 버튼 미노출) |
| S6. 교수자가 결과/통계를 확인한다 | O | QuizStats 구현 | 채점 완료 전 통계 접근 가능 여부 정책 미정 |
| S7. 교수자가 문제은행에서 문항을 관리한다 | O | QuestionBank 구현 | 문항 수정/삭제 시 연결된 퀴즈 영향도 처리 미정 |

### 주요 판정

- S4 (학습자 응시 화면) 부재는 프로토타입 범위로 양해 가능하나, S5 채점 저장이 없는 것은 Happy Path 단절로 Critical 이슈
- 퀴즈 목록(QuizList)에서 "채점하기" 버튼이 grading 상태인 퀴즈에만 노출되는 점은 적절하나, 상태 전이 트리거가 없어 상태가 고정됨

## 2. 예외 케이스 및 엣지 케이스 목록

### 2-1. 퀴즈 생성/편집 영역

| **우선순위** | **케이스** | **현재 처리** | **필요 처리** |
|---|---|---|---|
| Critical | 문항 0개 상태에서 발행 시도 | 버튼 활성화됨 (isFormValid는 title+날짜만 체크) | 문항 수 0이면 발행 차단 + 안내 메시지 |
| Critical | 배점 합계가 0점인 채로 발행 | 미처리 | 총점 0점 발행 차단 |
| High | 시작 일시 > 마감 일시로 설정 | 미처리 (UI 레벨 검증 없음) | 마감이 시작보다 빠른 경우 저장 차단 |
| High | 이미 발행된(published=true) 퀴즈에서 문항 삭제/수정 | 미처리 | 응시 중인 학생 있을 경우 수정 경고 또는 차단 정책 필요 |
| High | 복수 응시 허용 시 재응시 점수 정책 미선택 | 기본값으로 처리되나 UX 안내 없음 | "최고 점수 유지"를 기본값으로 명시하고, 변경 시 학생에게 소급 적용 여부 정책 필요 |
| Medium | 퀴즈 제목 중복 | 미처리 | 동일 코스 내 제목 중복 경고 (차단은 선택적) |
| Medium | 지각 제출 허용 체크 후 마감 기한 재설정 | 상태 유지 | 지각 제출 허용 기간(마감+N일) 별도 설정 필요 |
| Medium | 직접 추가 모달에서 선택지 없는 객관식 저장 | 미처리 | 객관식/복수선택 유형은 선택지 최소 2개 필수 검증 |
| Low | 퀴즈 제목 300자 초과 입력 | 미처리 | 최대 글자 수 제한 및 카운터 표시 |
| Low | 타임존 다른 교수/학생 간 응시 기간 표시 차이 | 미처리 | KST 기준 명시 또는 사용자 로컬 타임 대응 정책 |

### 2-2. 채점 영역

| **우선순위** | **케이스** | **현재 처리** | **필요 처리** |
|---|---|---|---|
| Critical | 채점 중 저장 없이 페이지 이탈 | 저장 버튼 없음, 경고 없음 | 채점 내용 임시 저장 또는 이탈 전 확인 모달 필요 |
| Critical | 동일 문항을 복수 채점자가 동시에 채점 | 미처리 | 락(Lock) 정책 또는 충돌 감지 후 마지막 저장 우선 정책 명시 |
| High | 단답형(short_answer) autoGrade가 mockData에서 'partial'로 정의되었으나 GradingDashboard에서는 autoGrade=false(수동)로 필터링됨 | 정합성 불일치 | short_answer의 자동채점 처리 정책 확정 필요 |
| High | 이미 채점된 문항을 교수자가 수정할 때 학생에게 알림 여부 | 미처리 | 재채점 플로우(현재 모달 존재) 확정 및 알림 정책 연결 필요 |
| High | 미제출 학생에 대한 점수 처리 (0점 자동 부여 vs 결시 처리 vs 별도 상태) | 미처리 | 미제출 상태의 점수 정책 명시 필요 |
| Medium | 부분 점수 허용 문항에서 0점 이하 입력 | 미처리 | 최솟값 0점 강제, 최댓값은 배점 초과 불가 처리 |
| Medium | 학생 중심 채점 모드에서 특정 학생만 수동 문항 일부가 채점된 상태 | 부분 처리 (미채점 표시) | 문항별 미채점 상태 시각화 + 해당 학생 총점 미확정 처리 |
| Low | 채점 완료 후 교수자가 문항 배점을 소급 변경 | 미처리 | 배점 변경 시 기채점 점수 재계산 여부 정책 필요 |

| **우선순위** | **케이스** | **현재 처리** | **필요 처리** |

## 3. 기능 정책 완결성 검토

**현재 구현 상태**: QuizCreate에서 최대 응시 횟수(1회~무제한) 선택 가능, 복수 응시 시 채점 방식(최고점수/최신점수/평균) 선택 가능

**정책 공백 항목**:

| **정책 항목** | **현재 상태** | **필요 정의** |
|---|---|---|
| 재응시 간격 제한 | 미정 | 재응시 가능 최소 간격(예: 즉시/1시간 후/다음날) 설정 여부 |
| 재응시 시 이전 답안 표시 여부 | 미정 | 이전 응시 답안을 보고 재응시 가능한지 여부 (Canvas 기본값: 비공개) |

### 3-2. 지각 제출 정책

**현재 구현 상태**: "마감 후 지각 제출 허용" 토글만 존재, 세부 정책 없음

**정책 공백 항목**:

| **정책 항목** | **현재 상태** | **필요 정의** |
|---|---|---|
| 지각 제출 허용 기간 | 미정 | 마감 후 몇 시간/일까지 허용할지 기간 설정 UI 필요 |
| 지각 제출 시 감점 정책 | 미정 | 원점수 vs 감점 적용(예: 10%/일 차감) 선택 여부 |
| 지각 제출 학생 구분 표시 | 미처리 | 채점 대시보드에서 지각 제출 학생 별도 표시 여부 |

### 3-3. 점수 공개 정책

**현재 구현 상태**: "제출 후 정답 공개" 토글만 존재

**정책 공백 항목**:

| **정책 항목** | **현재 상태** | **필요 정의** |
|---|---|---|
| 정답 공개 시점 | "제출 후" 단일 옵션 | 제출 직후 / 마감 후 / 채점 완료 후 / 교수자 수동 공개 중 선택 |

## 4. 상태 전이 정합성

### 현재 정의된 퀴즈 상태

```
draft (초안) → open (진행중) → grading (채점중) → closed (종료)
```

### 상태 전이 다이어그램

```
[draft]
  - 미발행 상태
  - 편집 가능
  - 학생 접근 불가
      ↓ 발행(publish)
[open]
  - 발행된 상태
  - 응시 기간 내 학생 응시 가능
  - 편집 제한 (발행 후 문항 변경 정책 필요)
      ↓ 마감일시 도달 (자동 전이)
[grading]
  - 응시 기간 종료
  - 수동채점 필요 문항 존재
  - 교수자 채점 진행 중
      ↓ 전체 채점 완료 (수동 또는 자동?)
[closed]
  - 채점 완료
  - 통계/결과 열람 가능
  - 점수 공개 여부에 따라 학생에게 결과 표시
```

### 상태 전이 정책 공백

| **항목** | **현재 상태** | **필요 정의** |
|---|---|---|
| draft → open 전이 트리거 | "저장 및 발행" 버튼 클릭 | 발행 전 체크리스트 확인(문항 수, 배점, 응시 기간 유효성) 후 전이 |
| open → grading 전이 트리거 | 미정 (mock 데이터 고정) | 마감 일시 도달 시 자동 전이 vs 교수자 수동 마감 중 선택 |
| grading → closed 전이 트리거 | 미정 | 전체 채점 완료 시 자동 전이 vs 교수자 수동 종료 중 선택 |
| open 상태에서 초안으로 되돌리기 | 미정 | 이미 응시한 학생이 있는 경우 되돌리기 차단 여부 |
| 자동채점만 있는 퀴즈의 grading 상태 스킵 | 미정 | 수동채점 문항이 없으면 마감 즉시 closed 전이 정책 확정 필요 |
| published=true이면서 status=draft인 상태 | mockData에서 발생 가능 | published 필드와 status 필드 간 정합성 정의 필요 (둘 중 하나로 통일 권장) |

### 데이터 모델 불일치 (Critical)

mockData.js에서 `published` 필드(boolean)와 `status` 필드(string)가 별도 존재한다. 예를 들어 `published=false, status='draft'`는 정합하나, `published=true, status='draft'`가 발생 가능한 구조다. 두 필드의 역할을 명확히 분리하거나 하나로 통일해야 한다.

**권고**: `status`를 단일 진실 공급원(SSOT)으로 사용하고 `published` 필드는 `status !== 'draft'`로 파생하는 방식 채택

## 5. 권한 설계 검토

### 현재 구현 상태

현재 프로토타입에는 역할(Role) 구분 자체가 없다. 모든 화면이 교수자 관점으로 고정되어 있으며, 로그인/인증 플로우도 없다.

### 필요 권한 설계 (역할 정의)

| **역할** | **정의** |
|---|---|

## 7. MVP 범위 및 우선순위 로드맵

### MVP1 완료 기준 (현재 프로토타입 대상)

MVP1 목표: 교수자가 퀴즈를 생성하고 채점 및 결과를 확인하는 핵심 플로우 검증

| **기능** | **현재 상태** | **Gap** |
|---|---|---|
| 퀴즈 목록 조회 및 상태 표시 | 완료 | - |
| 퀴즈 생성 (기본 정보 + 문항 구성) | 완료 | 저장 후 목록 반영 없음 |
| 퀴즈 편집 | 완료 | 저장 후 목록 반영 없음 |
| 문제은행 관리 | 완료 | 수정/삭제 UI는 있으나 동작 미연결 |
| 채점 대시보드 (문항 중심 + 학생 중심) | 완료 | 채점 저장 버튼 없음 |
| 퀴즈 통계 페이지 | 완료 | - |

### Critical 수정 항목 (MVP1 완료 전 필수)

| **ID** | **항목** | **근거** |
|---|---|---|
| C-01 | 채점 저장 기능 (저장 버튼 또는 자동저장) 미구현 | Happy Path 단절 |
| C-02 | 문항 0개 상태에서 발행 가능 | 데이터 정합성 위반 |
| C-03 | published 필드와 status 필드 이중 관리 | 상태 전이 정합성 오류 |
| C-04 | short_answer autoGrade 'partial' 정의 vs 수동채점 처리 불일치 | 채점 로직 오류 가능성 |

### MVP2 우선순위 로드맵

**목표**: 실제 LMS 연동 전 핵심 정책 구체화 및 학습자 경험 추가

#### Phase 2-A (High 우선)

| **기능** | **설명** | **우선순위 근거** |
|---|---|---|
| 학습자 응시 화면 | 문항 유형별 응시 UI, 타이머, 제출 확인 플로우 | 서비스 핵심 플로우 완성 |
| 퀴즈 상태 전이 자동화 | 마감일시 도달 시 open→grading 자동 전이 | 운영 효율 |
| 채점 저장 및 점수 공개 플로우 | 채점 완료 → 점수 공개 → 학생 결과 열람 | Happy Path 완성 |
| 지각 제출 허용 기간 설정 UI | 마감 후 허용 기간 및 감점 정책 적용 | 정책 공백 해소 |
| 발행 전 체크리스트 | 문항 수, 배점, 날짜 유효성 검증 + 확인 모달 | 교수자 실수 방지 |

#### Phase 2-B (Medium 우선)

| **기능** | **설명** | **우선순위 근거** |
|---|---|---|
| 조교 권한 설계 | 채점 권한 위임, 역할별 접근 제어 | 대학 LMS 운영 현실 반영 |
| 채점 기준표(Rubric) | 서술형 문항 채점 항목 등록 | 채점 일관성 확보 |
| 알림 시스템 | 마감 임박, 채점 완료, 점수 공개 알림 | 사용자 참여율 향상 |
| 정답 공개 세부 정책 | 공개 시점/항목 분리 설정 | Canvas LMS 표준 수준 |
| 문제은행 문항 수정 이력 | 변경 이력 및 연결 퀴즈 영향도 표시 | 데이터 무결성 |

#### Phase 2-C (Nice-to-have)

| **기능** | **설명** |
|---|---|
| AI 보조 채점 | 서술형/단답형 AI 초안 채점 제안 |
| 채점 일관성 분석 | 동일 문항 채점자 간 편차 통계 |
| 퀴즈 복사/템플릿 | 기존 퀴즈를 복사하여 신규 생성 |
| 문항 난이도 자동 추정 | 과거 응시 데이터 기반 난이도 레이블 |
| 응시 이상 행동 감지 | 탭 이탈 횟수, 동일 IP 중복 접속 탐지 |

### MVP 범위 외 (Out-of-scope)

| **항목** | **이유** |
|---|---|
| LMS 연동 (Canvas API, LTI) | 별도 개발 트랙 |
| 실시간 응시 모니터링 (proctoring) | 별도 솔루션 검토 필요 |
| 모바일 네이티브 앱 | 웹 반응형으로 대체 |
| 다국어 지원 | 현재 한국어 단일 서비스 |
