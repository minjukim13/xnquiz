## 1. 핵심 사용자 시나리오 (Happy Path) 정합성

### 시나리오 1 — 교수자: 퀴즈 생성 후 배포

| 단계 | 구현 현황 | 평가 |
|------|-----------|------|
| **퀴즈 제목 입력** | 필수 필드, 미입력 시 발행 버튼 비활성화 | 정상 |
| **주차/차시 선택** | 1~16주차, 1~4차시 선택 가능. 단, 필수 아님 | 주의 — 미선택 시 기본값 1/1로 저장됨 |
| **응시 기간 설정** | 시작/마감 datetime-local 입력, 필수 필드 | 정상 |
| **마감 후 지각 제출 허용 체크** | 체크박스 존재 | 정상 (정책 미정, 하단 이슈 참조) |
| **응시 시간 제한** | 30/60/90/120분 + 직접 입력(분) + 제한 없음 | 정상 |
| **최대 응시 횟수** | 1/2/3/무제한 선택 | 정상 |
| **복수 응시 채점 방식** | 최고 점수 유지 / 최신 점수 유지 / 평균 점수 | 정상 |
| **선택지/문항 무작위** | 토글 제공 | 정상 |
| **제출 후 정답 공개** | 토글, 기본값 true | 정상 |
| **퀴즈 안내사항** | 자유 텍스트, 기본값 제공 | 정상 |
| **문항 추가 (문제은행)** | 은행 필터 + 검색 후 추가. 중복 추가 방지 로직 있음 | 정상 |
| **문항 직접 추가** | AddQuestionModal 통해 유형/점수/텍스트 입력 | 정상 |
| **발행하기** | 유효성 검사(제목+기간) 통과 시 mockQuizzes 배열에 push 후 목록으로 이동 | 부분 정상 — 임시저장 버튼은 UI만 존재, 실제 동작 없음 |

**시나리오 1 종합 평가**: 기본 흐름은 동작하나, 임시저장 미구현, 발행 시 상태가 무조건 'open'으로 고정되어 초안 상태로 저장 불가.

### 시나리오 2 — 교수자: 채점 대시보드 활용

| 단계 | 구현 현황 | 평가 |
|------|-----------|------|
| **채점 대시보드 진입** | 퀴즈 목록에서 '채점하기' 버튼 — status='grading'인 퀴즈에만 노출 | 정상 |
| **문항 중심 채점** | 주관식 문항 목록 → 학생 응답 목록 → 점수 입력 → 저장 | 정상 |
| **학생 중심 채점** | 학생 목록 → 전체 문항 답안 카드 → 문항별 점수 입력 | 정상 |
| **점수 저장** | localStorage에 xnq_manual_grades 키로 저장, 총점 자동 재계산 | 정상 |
| **채점 진행률 표시** | 퀴즈 정보 카드 내 프로그레스바 | 정상 |
| **엑셀 일괄 채점** | 버튼 클릭 시 모달 표시 | UI만 구현, 실제 파일 Import/Export 미구현 |

| **채점 완료 후 상태 전환** | 수동으로 상태 변경하는 버튼/플로우 없음 | 누락 — Critical |

**시나리오 2 종합 평가**: 핵심 채점 기능(문항/학생 중심 모두)은 동작하나, 채점 완료 후 상태를 'closed'로 전환하는 명시적 액션이 없음.

## 2. 예외 케이스 및 엣지 케이스 목록

| 번호 | 케이스 | 현재 처리 | 심각도 |
|------|--------|-----------|--------|
| **E-01** | 마감 일시가 시작 일시보다 이른 경우 | 유효성 검사 없음, 저장 허용 | **Critical** |
| **E-02** | 문항이 0개인 퀴즈 발행 | 유효성 검사 없음, 발행 허용 | **Critical** |
| **E-03** | 채점 점수 입력란에 소수점 입력 (예: 8.5점) | min/max 검증만 있고 정수 강제 없음 | **High** |
| **E-04** | 채점 점수가 배점 초과 입력 | `Number(score) > question.points`로 저장 버튼 비활성화 | 정상 |
| **E-05** | 미제출 학생이 채점 대상 목록에 노출되는지 여부 | `s.submitted === true` 필터로 제출자만 표시 | 정상 |
| **E-06** | URL 직접 접근 (예: /quiz/999/grade) | mockQuizzes.find 실패 시 `?? mockQuizzes[0]`로 fallback — 첫 번째 퀴즈 채점 페이지가 열림 | **High** |
| **E-07** | 동일 학생 동일 문항 점수 이중 저장 | 덮어쓰기 방식, localStorage key는 `quizId_studentId_questionId`로 고유 | 정상 |
| **E-08** | 문항 추가 없이 문제은행 모달에서 아무 것도 선택 안 하고 닫기 | 별도 처리 없음, 정상 취소 | 정상 |
| **E-09** | 복수 응시 시 서술형 문항 재채점 정책 | 이전 채점 결과 유지 여부 정책 없음 | **High** |
| **E-10** | 지각 제출 허용 체크 후 실제 지각 제출 처리 | 체크박스만 존재, 지각 제출 시 점수 차감 여부 등 정책 미구현 | **High** |
| **E-11** | 문항 배점 합계가 총점과 다른 경우 발행 | totalPoints는 questions 배열 합산으로 자동 계산됨 | 정상 |
| **E-12** | 브라우저 새로고침 후 퀴즈 생성 폼 데이터 유지 여부 | localStorage 저장 없음, 새로고침 시 폼 초기화 | Medium |
| **E-13** | 채점 대시보드에서 open/draft 퀴즈 접근 (URL 직접 입력) | status 체크 없음, 채점 대시보드 정상 노출됨 | **High** |
| **E-14** | 검색 결과 0건 시 빈 상태 | "검색 결과가 없습니다" 메시지 표시 | 정상 |
| **E-15** | mockQuestions가 특정 퀴즈에 종속되지 않고 전역 공유됨 | /quiz/2/grade, /quiz/3/grade 등 접근 시 ID=1 기준 문항이 표시됨 | **Critical** |
| **E-16** | 단답형 자동채점 정답 대소문자 구분 | `toLowerCase()` 포함 비교로 처리 | 정상 |
| **E-17** | 엑셀 일괄 채점 업로드 후 검증 실패 처리 | 미구현 (모달 UI만 존재) | Medium |
| **E-18** | 문제은행에서 삭제한 문항이 기존 퀴즈에 포함된 경우 처리 | 미정의, 삭제 전 경고 UI 아이콘은 임포트됨 | High |

## 4. 상태 전이 다이어그램 정합성

### 현재 퀴즈 상태 (STATUS_CONFIG 기준)

```
draft (초안) → open (진행중) → grading (채점중) → closed (종료)
```

### 상태별 코드 분석

| 상태 | 전환 트리거 | UI 액션 | 이슈 |
|------|------------|---------|------|
| **draft** | 퀴즈 생성 시 초기 상태 (예정) | 편집 가능, 채점/통계 접근 시 URL로는 가능 | QuizCreate에서 발행 시 바로 'open'으로 생성됨 — draft로 저장 불가 |
| **open** | 발행 직후 | 편집/통계 접근 가능, 채점하기 버튼 없음 | 응시 기간 도래 여부와 무관하게 open 표시됨 |
| **grading** | mockData에 수동 고정 | 채점하기 버튼 활성화 | 자동 전환 로직 없음 — 마감 후 자동으로 grading 전환 미구현 |
| **closed** | mockData에 수동 고정 | 결과 보기 버튼만 노출, 편집 버튼 없음 | 채점 완료 후 closed로 전환하는 교수자 액션 없음 |

### 누락된 상태 전이

- **open → grading**: 마감 일시 도래 시 자동 전환 미구현
- **grading → closed**: 교수자가 "채점 완료" 확정하는 액션 미구현
- **closed → grading**: 점수 수정이 필요한 재채점 시 역전환 미구현
- **draft → open**: 발행 액션은 있으나, 발행 전 미리보기(Preview) 단계 없음

### Canvas LMS 대비 차이점

Canvas LMS는 Published/Unpublished 이분법에 Available Date/Until Date를 조합하여 실질적인 상태를 제어함. XN Quizzes는 4단계 상태를 명시했으나 전환 트리거가 수동 데이터 변경에 의존하고 있어 실서비스 전환 시 상태 관리 전략 재설계 필요.

## 5. 권한 설계

### 현재 구현 현황

현재 프로토타입은 단일 교수자 관점으로만 구현되어 있으며, 역할 기반 접근 제어(RBAC)가 전혀 없음.

| 기능 | 교수자 | 조교 | 학습자 | 현재 구현 |
|------|--------|------|--------|-----------|
| 퀴즈 생성/편집 | O | 조건부 | X | 권한 구분 없음 |
| 채점 | O | O | X | 권한 구분 없음 |
| 성적 조회 (전체) | O | O | X | 권한 구분 없음 |
| 성적 조회 (본인) | - | - | O | 학습자 화면 없음 |
| 문제은행 접근 | O | 조건부 | X | 권한 구분 없음 |
| 통계 조회 | O | O | X | 권한 구분 없음 |
| 답안 확인 (본인) | - | - | O | 학습자 화면 없음 |

### 누락된 역할별 페이지

- **학습자 응시 화면** (퀴즈 풀기 페이지) — 전혀 없음
- **학습자 결과 확인 화면** (본인 점수/정답 확인) — 없음
- **조교 권한 설정** — 없음
- **로그인/인증** — 없음

## 8. 통합 이슈 목록

| 번호 | 이슈 | 영역 | 심각도 | 수정 권고 |
|------|------|------|--------|-----------|
| **I-01** | mockQuestions가 특정 퀴즈에 종속되지 않고 모든 채점 페이지에서 동일하게 표시됨 | 데이터 모델 | **Critical** | 퀴즈별 문항 데이터 분리 필요 |
| **I-02** | 채점 완료 후 퀴즈 상태를 closed로 전환하는 액션 없음 | 상태 전이 | **Critical** | "채점 완료" 버튼 및 상태 전환 처리 추가 |
| **I-03** | 마감 일시 < 시작 일시 허용 — 유효성 검사 없음 | 생성 폼 | **Critical** | 클라이언트 유효성 검사 추가 |
| **I-04** | 문항 0개 퀴즈 발행 허용 | 생성 폼 | **Critical** | questions.length > 0 조건 추가 |
| **I-05** | URL 직접 입력으로 임의 quizId 접근 시 첫 번째 퀴즈 데이터 노출 | 보안/라우팅 | **High** | ID 불일치 시 404 또는 목록 리다이렉트 |
| **I-06** | status='grading'이 아닌 퀴즈도 URL로 채점 대시보드 접근 가능 | 보안/라우팅 | **High** | status 체크 후 접근 차단 |
| **I-07** | 지각 제출 허용 체크 후 실제 적용 정책 미정의 | 정책 | **High** | 허용 기간, 감점 방식 정책 확정 필요 |
| **I-08** | 복수 응시 채점 방식 선택 후 실제 적용 로직 없음 | 정책/구현 | **High** | 복수 응시 이력 데이터 모델 + 채점 로직 구현 |
| **I-09** | 문제은행 데이터 새로고침 시 초기화 (localStorage 미사용) | 영속성 | **High** | 추가/편집/삭제 결과 localStorage 저장 또는 API 연동 |
| **I-10** | 채점 소수점 점수 허용 여부 미명세 | 정책 | **High** | 정수만 허용 시 step="1" 및 정수 검증 추가 |
| **I-11** | 퀴즈 생성 시 임시저장 버튼 동작 없음 | 구현 | **High** | localStorage 활용 임시저장 구현 |
| **I-12** | open/draft 퀴즈에서도 통계 페이지 접근 가능 | UX/정책 | **Medium** | 제출자 없을 경우 빈 상태 안내 + 접근 제한 정책 결정 |

## 자체 리뷰

**잘 구현된 부분**
- 교수자 중심 채점 워크플로우(문항 중심/학생 중심 이중 모드)는 Canvas LMS의 SpeedGrader에 근접한 UX를 구현했으며 프로토타입으로서 설득력이 높음
- localStorage 기반 점수 저장 및 총점 자동 재계산 로직은 실서비스 전환 시 API 교체만 하면 되는 구조로 잘 설계됨
- 퀴즈 통계 화면의 지표 구성(평균/표준편차/p27·p73/시간압박 비율)은 교육 현장 맥락에서 실질적인 인사이트를 제공함

**핵심 리스크**
- 퀴즈 ID와 문항 데이터가 분리되지 않아 멀티 퀴즈 환경에서 데이터 혼용 문제가 발생함 (I-01 — Critical)
- 상태 전이 자동화가 없어 실서비스에서는 교수자가 마감 여부를 수동으로 추적해야 하는 운영 부담이 생김 (I-02 — Critical)
- 학습자 화면이 전혀 없어 현재는 교수자 툴로만 기능하며, 실서비스 전환 시 학습자 UX 설계가 별도 스프린트 단위로 필요함
