# 사용자 시나리오 문서

**작성일**: 2026-04-28
**대상**: XN Quizzes (LearningX 연동 퀴즈 모듈) MVP1
**용도**: 제품 기획·디자인·개발·QA 공통 참조 (페르소나·흐름·감정·페인포인트)
**관련 문서**: `docs/QA/사용자시나리오테스트스크립트.md` (테스트 실행용), `PM2_기획검토보고서.md` (정책 잔존 이슈), `docs/XN_Quiz_vs_Canvas_UX비교.md`

---

## 1. 문서 목적

- 제품에 등장하는 사용자가 **누구이고, 언제, 왜, 어떻게** XN Quizzes 를 사용하는지 정리
- 단순한 기능 나열이 아니라 **맥락·동기·감정**을 함께 담아 설계 의사결정의 근거로 활용
- 신규 합류자가 이 문서 하나로 도메인을 이해할 수 있는 수준의 깊이 유지

### 작성 원칙
- 실명/실제 학번/실제 이메일 사용 금지 (CLAUDE.md 정책)
- 모든 시나리오는 시드 데이터 기준 (학생 A~L, `*@xn.test`)
- 정책 미확정 영역은 **[정책 결정 필요]** 표시
- 페인포인트는 **현재 상태 / 사용자 영향 / 우선순위** 함께 기재

## 2. 페르소나

### P1. 김교수 (40대, 부교수, 평균 강좌 2개 운영)
- **소속**: 일반 4년제 대학 인문계열
- **디지털 친숙도**: 보통 (LMS 사용 5년차, 엑셀·PPT 능숙, 신규 도구 학습은 시간 소요)
- **퀴즈 운영 패턴**: 학기 중 주 1회 출석퀴즈, 중간/기말 직전 모의고사 1회
- **목표**: 빠르게 출제하고, 채점 부담을 줄이고, 학생 성취도를 학기말 보고서에 활용
- **주요 페인포인트**:
  - 매학기 같은 퀴즈를 다시 만드는 게 번거로움
  - 서술형 채점에 시간이 너무 걸려서 과목당 2일 소요
  - 정답을 잘못 지정한 걸 발행 후 발견했을 때 어떻게 정정해야 할지 막막함

### P2. 이학생 (20대, 학부 2학년, 1주에 평균 4개 강좌 수강)
- **디지털 친숙도**: 높음 (모바일 우선, LMS 앱 사용)
- **응시 패턴**: 마감 임박 응시 비율 높음, 출석퀴즈는 강의실에서 즉시 응시
- **목표**: 마감 안 놓치기, 점수와 정답 빠르게 확인하기
- **주요 페인포인트**:
  - 응시 도중 와이파이 끊기면 답안 날아갈까 불안
  - 제출 후 점수만 보이고 어디 틀렸는지 안 알려줘서 학습이 안 됨
  - 재응시 가능한 퀴즈인지 응시 전엔 잘 안 보임

### P3. 박조교 (대학원생, 강좌당 평균 60명 학습자 관리)
- **권한**: 현재 별도 역할 미구현 (교수자 권한 공유로 운영) **[정책 결정 필요]**
- **목표**: 교수자 대신 채점 1차 진행, 이상 응시 모니터링
- **주요 페인포인트**:
  - 교수자 계정 공유 사용 시 책임 소재 불명확
  - 누가 언제 어떤 점수를 매겼는지 이력 추적 어려움

### P4. 정운영자 (학교 LMS 담당자, 50개 강좌 모니터링)
- **권한**: ADMIN (교수자 권한 포함, 메모리 `feedback_admin_includes_professor`)
- **목표**: 학기 운영 중 발생하는 문제(부정행위, 시스템 오류, 점수 분쟁) 1차 대응
- **주요 페인포인트**:
  - 교수자 문의("왜 학생 점수가 0점이에요?") 발생 시 원인 추적 어려움
  - 동시 응시·접속 폭주 시점에 시스템 모니터링 수단 부족

## 3. 핵심 사용자 시나리오

### SC-01. 김교수, 학기 첫 출석퀴즈를 만든다 [Critical]

**페르소나**: P1 (김교수)
**트리거**: 개강 첫 주 강의 직후, 다음 주 출석퀴즈 출제 필요
**사전 조건**: LearningX 강좌 개설 완료, XN Quizzes 메뉴 진입 가능

**흐름**

| Step | 김교수 행동 | 시스템 응답 | 김교수 감정 |
|---|---|---|---|
| 1 | LearningX 강좌 → 퀴즈 메뉴 클릭 | 빈 퀴즈 목록 + "퀴즈 만들기" CTA | "어디부터 시작해야 하지?" |
| 2 | "퀴즈 만들기" 클릭 | 2탭 단일 화면(`기본 정보` / `문항`), 기본 정보 탭 활성 | "탭 두 개라 부담은 덜한데, 입력 항목이 많네" |
| 3 | 제목 "1주차 출석퀴즈", 설명 입력 | 폼 입력 진행 | |
| 4 | 시간제한 10분, 재응시 0회, 지각허용 OFF | 같은 탭 안에서 시간/재응시/지각/성적 공개 정책 모두 입력 | "이 정도면 부정행위 방지 충분?" |
| 5 | `문항` 탭으로 전환, 4지선다 5문항 추가 | 문항별 카드 등록 | "문항 5개에 시간이 꽤 드네" |
| 6 | 발행 버튼 클릭 | 확인 모달 → 발행 완료 | "이제 학생들이 볼 수 있나?" |
| 7 | 퀴즈 목록/상세 더보기 메뉴 → "미리보기" 진입 | 학생 시점 응시 화면(`?preview=true`), "정답 함께 표시" 토글 제공 | "학생 시점 그대로 검증 가능, 안심" |

**결과**: 퀴즈 상태 "진행중", 학생 홈에 노출, 응시 가능

**페인포인트 / 개선 기회**

| # | 현재 상태 | 사용자 영향 | 우선순위 |
|---|---|---|---|
| F-01 | 미리보기 진입 동선이 더보기 메뉴(⋯) 안쪽에 위치, 발견성 낮음 | 미리보기 기능을 인지 못한 교수가 본인 학생 계정으로 응시해 확인하는 사례 발생 가능 | Medium |
| F-02 | 출제 화면(생성/편집)은 자동저장 없음, 명시적 임시저장만 동작 (응시 화면은 30초 자동저장 구현됨) | 출제 중 브라우저 닫으면 손실 | Medium |
| F-03 | 시간제한·재응시·성적 공개 정책의 의미 안내 부재 | "이게 뭔지 모르겠다" 1차 지원 문의 발생 가능 | Medium |

**관련 정책**: 발행 = `draft → open` 전이 (PM2 상태 전이 다이어그램)

### SC-02. 김교수, 지난 학기 퀴즈를 복사해서 다시 쓴다 [High]

**페르소나**: P1
**트리거**: 같은 과목을 다시 강의, 시간 절약
**사전 조건**: 지난 학기 발행된 퀴즈 1개 이상

**흐름**

| Step | 행동 | 시스템 응답 | 감정 |
|---|---|---|---|
| 1 | 퀴즈 목록 → 지난 학기 퀴즈의 더보기 메뉴 → 복사 | 복사 모달 | "복사 한 번에 끝나면 좋겠다" |
| 2 | 새 제목 "1주차 출석퀴즈 (2026)" 입력 | 임시저장 상태 신규 퀴즈 생성 | |
| 3 | 복제본 편집 진입, 1개 문항 정답 수정 | 원본 정답은 유지 (독립성) | "원본 망가지지 않게 잘 분리됐네" |
| 4 | 시간제한만 5분으로 변경 후 발행 | 신규 회차 발행 완료 | "다음 학기엔 더 빠르겠다" |

**결과**: 신규 퀴즈가 별개 ID로 등록, 학생들에게 새 퀴즈로 노출

**페인포인트**

| # | 현재 상태 | 사용자 영향 | 우선순위 |
|---|---|---|---|
| F-04 | 복사 시 응시 기록은 미복제(정상)이지만 "복사본임"을 명시하는 배지 없음 | 목록에서 원본/복사본 구분 어려움 | Low |
| F-05 | 학기·연도 필드 부재로 같은 제목 누적 시 혼란 | 검색·정렬 비효율 | Medium |

### SC-03. 김교수, 발행 후 정답 오류를 발견하고 정정한다 [Critical]

**페르소나**: P1
**트리거**: 학생 채점 완료된 퀴즈에서 본인이 정답을 잘못 지정한 것을 발견
**감정 시작점**: "어떡하지... 학생들이 항의하면 어떻게 설명하지"

**흐름**

| Step | 행동 | 시스템 응답 | 감정 |
|---|---|---|---|
| 1 | 채점 대시보드에서 이상한 정답률 1번 문항 발견 | 정답률 8% 표시 | "왜 다 틀렸지?" |
| 2 | 편집 진입 → 1번 문항 정답 변경 | 응시 기록 있음 경고 배너 | "건드려도 되나?" 망설임 |
| 3 | 저장 시도 | RegradeOptionsModal: 4가지 옵션 제시 | "옵션이 있구나, 다행" |
| 4 | "이전+신규 인정" 선택 | 기존 정답자 점수 유지, 신규 정답자 가산 | "공평하다" 안도감 |
| 5 | 채점 대시보드 재진입 | 재채점 로그(시각/옵션/대상수) 표시 | "기록 남으니까 이의제기 와도 설명 가능" |

**결과**: 학생 점수가 정책에 맞게 일괄 갱신, 변경 이력 추적 가능

**페인포인트**

| # | 현재 상태 | 사용자 영향 | 우선순위 |
|---|---|---|---|
| F-06 | 4개 옵션의 차이를 본문 외에 시각적으로 비교하기 어려움 | 잘못된 옵션 선택 위험 | High |
| F-07 | 학생에게 "재채점이 적용됐다"는 알림 없음 | 학생이 점수 변동을 모름 | High |
| F-08 | 재채점 후 영향받은 학생만 필터하는 기능 부재 | 일일이 확인 어려움 | Medium |

**관련 메모리**: PM4 보고서 M-03 해결(`0138d85`, `e507115`)

### SC-04. 김교수, 서술형 답안을 채점한다 [High]

**페르소나**: P1
**트리거**: 마감 직후, 50명분 서술형 채점 시작
**감정 시작점**: "이거 끝내려면 하루 종일 걸리겠네"

**흐름**

| Step | 행동 | 시스템 응답 | 감정 |
|---|---|---|---|
| 1 | 채점 대시보드 진입 → 미채점 필터 | 미채점 학생 50명 리스트 | "양이 많네" |
| 2 | 학생 A 선택 | 답안 + 자동채점 결과 + 서술형 입력 영역 | |
| 3 | 서술형 답안 검토 → 점수/피드백 입력 | 즉시 저장, 학생 합계 갱신 | "한 명 끝났다" |
| 4 | 다음 학생으로 이동 (단축키 미구현) | 마우스로 클릭 | "키보드로 빨리 넘기면 좋겠다" |
| 5 | 50명 채점 완료 후 "최종 확정" 시도 | **현재 흐름 부재 (E-08)** | "이게 진짜 다 채점된 건지 모르겠다" |

**결과**: 개별 채점은 저장되나 "전체 확정" 단계 없음

**페인포인트**

| # | 현재 상태 | 사용자 영향 | 우선순위 |
|---|---|---|---|
| F-09 | 학생 간 빠른 이동 단축키 부재 (마우스 클릭 전용, `onKeyDown` 핸들러 없음) | 채점 시간 증가 | High |
| F-10 | 채점 완료 확정 흐름 부재 (E-08). 개별 채점은 저장되나 "전체 확정" 단계 없음 | "다 끝났다" 명확한 종료 신호 없음 | Critical |
| F-11 | 루브릭 필드는 문항 작성 시 입력 가능(essay 유형), 다만 동일/유사 답안 일괄 채점(같은 답안에 일괄 점수 부여) 기능 부재 | 비슷한 답안에 같은 점수 매번 수동 입력 | Medium |
| F-12 | 학생에게 점수 공개 시점 제어 부재 | 채점 도중 점수가 노출될 가능성 | High |

**[정책 결정 필요]**: 채점 완료 확정 = 즉시 학생에게 공개? 별도 공개 버튼?

### SC-05. 이학생, 마감 1시간 전 응시한다 [Critical]

**페르소나**: P2 (이학생)
**트리거**: 알림이나 캘린더 보고 마감 임박 인지
**감정 시작점**: 약간의 조급함, "마감 안 놓쳐서 다행"

**흐름**

| Step | 행동 | 시스템 응답 | 감정 |
|---|---|---|---|
| 1 | LearningX 모바일 → 강좌 → 퀴즈 진입 | 퀴즈 카드 (남은 시간, 시간제한, 재응시 표시) | "남은 시간 보이니까 안심" |
| 2 | 응시 시작 클릭 | 카운트다운 시작, 1번 문항 표시 | "긴장된다" |
| 3 | 4지선다 답안 선택 | 자동저장 (localStorage) | |
| 4 | 갑자기 모바일 데이터 끊김 | 화면은 유지, 답안은 로컬에 저장 | "어어 어떡하지" |
| 5 | 2분 후 와이파이 재연결 | 새로고침 시 답안 복원 | "다행이다 살았다" |
| 6 | 마지막 문항 답하고 제출 | 결과 화면 즉시 노출, 자동채점 점수 표시 | "잘 봤네/망했네" |
| 7 | 어디 틀렸는지 보려고 함 | **상세 결과 화면 부재 (MVP2)** | "왜 틀렸는지 모르면 다시 못 풀잖아" 답답함 |

**결과**: 자동채점은 즉시, 서술형은 교수자 채점 후 점수 갱신

**페인포인트**

| # | 현재 상태 | 사용자 영향 | 우선순위 |
|---|---|---|---|
| F-13 | 학생용 결과 상세(정오답·해설) 화면 부재 | 학습 효과 저하 | Critical |
| F-14 | 카운트다운 외 "남은 시간 경고"(예: 1분 전) 없음 | 시간 관리 어려움 | Medium |
| F-15 | 제출 후 "정상 제출됨" 명시적 확인 화면 약함 | 제출 여부 불안 | Medium |
| F-16 | 모바일 환경에서 카운트다운 백그라운드 동작 검증 안 됨 | 화면 끄면 시간 멈출 수 있음 | High |

### SC-06. 이학생, 재응시 가능한 퀴즈를 다시 푼다 [High]

**페르소나**: P2
**트리거**: 1차 응시 점수가 낮아서 재응시 시도
**감정 시작점**: "이번엔 잘 보자"

**흐름**

| Step | 행동 | 시스템 응답 | 감정 |
|---|---|---|---|

## 4. 곁다리 / 예외 시나리오

### SC-E1. 마감 직전 제출 경합 [정책 결정 필요]
- **상황**: 이학생이 마감 1초 전 제출 버튼 클릭, 도달은 마감 1초 후
- **현재 동작**: 지각 허용 OFF 시 처리 정책 불명확 (E-04, PM2)
- **결정 필요**: (a) 클라이언트 클릭 시각 기준 (b) 서버 도달 시각 기준 (c) 일정 buffer 허용

### SC-E2. 다중 탭 동시 응시 [정책 결정 필요]
- **상황**: 이학생이 같은 퀴즈를 탭 2개에서 동시 시작
- **현재 동작**: 둘 다 진입, 마지막 입력이 덮어씀
- **결정 필요**: 단일 세션 강제? 경고만?

### SC-E3. accessCode 보호 퀴즈
- **상황**: 김교수가 강의실 출석퀴즈에 코드 "1234" 설정
- **현재 동작**: 필드 저장만 됨, 검증 로직 부재 (E-14)
- **영향**: 강의실 외 응시 차단 불가

### SC-E4. 임시저장 중 브라우저 강제 종료
- **상황**: 김교수가 문항 작성 중 노트북 배터리 방전
- **현재 동작**: 출제 화면(생성/편집)은 명시적 임시저장 클릭 시점 외 데이터 손실. 응시 화면은 이미 30초 자동저장(`AUTOSAVE_INTERVAL_MS=30_000`) 동작 중
- **개선 기회**: 출제 화면에도 응시와 동일한 30초 자동 임시저장 적용

### SC-E5. 운영자가 학생 시점으로 화면 확인
- **상황**: 정운영자가 "학생에게 어떻게 보이나" 확인 필요
- **현재 동작**:
  - RoleToggle 로 학생 전환 가능 (메모리 `feedback_admin_includes_professor` 권한 정책)
  - 퀴즈 목록/상세의 "미리보기"(`?preview=true`)는 학생 시점 화면을 띄우면서 실제 응시 기록을 남기지 않음 (`isPreview` 분기로 attempt 세션·activity 로그·서버 제출 모두 차단)
- **개선 기회**: 미리보기 진입 동선이 더보기 메뉴(⋯) 안쪽에 있어 발견성 낮음, 상단 노출 검토

### SC-E6. 학생이 마감 후 점수 확인
- **상황**: 이학생이 학기말, 학기 중 본 퀴즈 점수 다시 확인
- **현재 동작**: 학생용 결과 상세(`ResultModal`) 화면은 구현됨. 단 `oneTimeResults` 활성 시 1회 조회 후 응답·정답 재조회 차단됨 (점수만 노출). 또한 마감 후 결과 재진입 동선(퀴즈 목록 → 결과 다시 보기 등)은 별도 검증 필요
- **영향**: oneTimeResults 활성 퀴즈는 학습 회고 불가, 성적 이의제기 시 본인 답안 확인 어려움

## 8. 자체 리뷰

**강점**
- 페르소나 4명, 시나리오 8개 + 예외 6개로 핵심 흐름은 모두 커버
- 감정선/페인포인트 명시로 디자인·기획 의사결정 근거 활용 가능
- PM2/PM4 보고서, 메모리 정책과 연동되어 단일 진실 공급원(SSOT) 역할

**약점**
- P3(조교) 권한 정책 미확정으로 시나리오 빈약 — 정책 결정 후 보강 필요
- 모바일 우선 시나리오는 SC-05 외 부족 — 별도 모바일 사용 시나리오 보강 권고
- LearningX 외 플랫폼(Canvas 직접 연동 등) 시나리오는 LTI POC 진행 후 추가

**권고**
- 본 문서는 분기 단위로 갱신 (정책 변경/MVP 단계 전환 시)
- 페르소나 인터뷰(실제 교수자 1~2명) 후 SC-01, SC-04 보강 시 신뢰도 상승
- SC-05 의 모바일 데이터 끊김 케이스는 실측 검증 필요
