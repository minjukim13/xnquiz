## 1. 각 PM 보고서 요약

### PM1 — 디자인 검토

indigo-600 기반의 모던 화이트 모드가 전반적으로 일관되게 적용되어 있고, `.btn-primary / .card-flat / .input` 등 컴포넌트 토큰 구조도 잘 갖춰져 있다. 다만 토큰이 정의되어 있음에도 인라인 스타일이 전 파일에 걸쳐 혼용되어 시각적 불일치가 발생한다. 접근성 측면에서 아이콘 전용 버튼에 `aria-label`이 전무하고(Critical), 삭제 확인 모달이 없어 즉시 삭제가 실행되는 점(Critical)이 즉시 수정이 필요한 핵심 이슈다. 고객사 요구사항(난이도 배지 UI, 복수 문제은행 선택, 랜덤 출제 설정 패널, DnD 순서 변경)에 해당하는 UI는 현재 전혀 구현되지 않았다.

### PM2 — 서비스 기획 검토

퀴즈 생성-채점-결과 확인의 Happy Path 흐름은 부분적으로 구현되어 있으나, 퀴즈 상태 전이 규칙(open→grading→closed)이 자동인지 수동인지 불명확하고 성적 공개 단계가 채점 완료와 분리되지 않는 정책적 공백이 있다. 핵심 리스크는 두 가지다. 응시 횟수 초과 차단 로직이 전혀 구현되지 않아 설정값과 무관하게 무제한 응시가 가능하고, 사용 중인 문제은행 삭제 시 소속 문항이 즉시 삭제되어 퀴즈 데이터 손실로 이어진다. 고객사 핵심 요구사항인 난이도/그룹 기반 랜덤 출제는 데이터 모델 자체부터 재설계가 필요한 구조적 누락이다.

### PM3 — 개발/성능 검토

번들 크기에서 xlsx(429 kB raw) 청크와 recharts(351 kB) 청크가 두드러지나, 두 청크 모두 lazy loading으로 분리되어 초기 로딩에는 미포함된다. Critical 코드 이슈는 두 가지다. `mockData.js`에 자동채점 정답 데이터가 클라이언트 번들에 그대로 노출되어 있고, `QuizCreate/QuizList`에서 ES 모듈 배열(`mockQuizzes`)을 직접 변이(push/인덱스 교체)하여 React 불변성 원칙을 위반하며 새로고침 시 상태가 소멸한다. API 전환 시 교체 대상 파일과 함수가 명확하게 식별되어 있으며, 전역 에러 바운더리 미존재로 런타임 오류 발생 시 흰 화면(White Screen of Death)이 발생할 수 있다.

### PM4 — QA 테스트

기능/성능/보안 총 48개 테스트 항목 중 PASS 26건, FAIL 12건, REVIEW 10건이다. 고객사 요구사항 미충족 항목이 Critical 3건(문제명 변경 미반영, 그룹별 랜덤 출제 미구현, 엑셀 업로드 난이도/그룹 컬럼 미파싱)이며, URL 직접 접근 시 인증/권한 체크가 없어 채점 대시보드와 통계 화면에 누구나 접근 가능한 보안 Critical이 별도로 존재한다. 임시저장 버튼이 클릭 시 아무 동작도 하지 않고(High), 복수 탭 동시 채점 시 localStorage가 덮어써지는 데이터 충돌 이슈(High)도 확인됐다.

### PM5 — PO 스펙/우선순위

MVP2의 최우선 목표는 문항 난이도/그룹 메타데이터 추가이며, 이것이 복수 은행 랜덤 출제(2-1) 및 엑셀 컬럼 확장(1-3)의 선행 조건이다. 부분 점수 정책, 단답형 정답 판정 옵션, 주차별 퀴즈 목록, 시험 복제/이관이 그 뒤를 따른다. IP 제한/TrustLock/멀티미디어 첨부 등 서버 인프라가 필요한 기능은 Out-of-scope로 명시됐다. 6개의 미결 정책(채점 완료 후 재응시 처리, 게시 중 정답 변경, 게시 취소 허용 여부, 부분 점수 감점, 주차별 뷰 기본값, 문항 삭제 시 퀴즈 처리)이 정의되었으며 PM6 이관이 권고됐다.

## 3. Critical 이슈 강조 섹션

### CR-01. 아이콘 전용 버튼 aria-label 전무

**관련 PM**: PM1
**재현 시나리오**: 문제은행 화면에서 Trash2(삭제) / Edit2(편집) / X(닫기) 버튼을 스크린리더로 탐색 → 버튼 목적 불명
**영향도**: 보조기기 사용자 전체 접근 불가. 접근성 법규(WCAG 2.1 AA) 위반 수준
**수정 권고안**: 모든 아이콘 전용 버튼에 `aria-label="문항 삭제"`, `aria-label="편집"` 등 추가. 작업 규모: 전 파일 30분 내외

---

### CR-02. 삭제 확인 모달 없음

**관련 PM**: PM1, PM2, PM4 (3개 PM 공통 지적)
**재현 시나리오**: 퀴즈에서 사용 중인 문제은행의 삭제 버튼 클릭 → 확인 없이 즉시 삭제 → 해당 은행 소속 전체 문항 연쇄 삭제 → 퀴즈 내 문항 참조 손실
**영향도**: 교수자 실수로 강의 자료 전체 소실 가능. 데이터 복구 불가 (프로토타입 수준)
**수정 권고안**: deleteBank / deleteQuestion 호출 전 확인 모달 추가. 퀴즈 사용 중 은행 삭제 시 "현재 N개 퀴즈에서 사용 중. 삭제하면 해당 퀴즈에 영향이 생깁니다" 경고 표시

---

### CR-03. 고객사 핵심 기능 3종 미구현

**관련 PM**: PM2, PM4, PM5 (3개 PM 공통 지적)
**항목**:
- 문항 난이도/그룹 메타데이터 (Question 데이터 모델에 `difficulty`, `groupTag` 필드 자체 없음)
- 그룹별 출제 수 + 난이도별 배점 랜덤 출제 (QuizCreate에 관련 UI 전무)
- 엑셀 업로드 난이도/그룹 컬럼 파싱 (`parseExcelOrCsv`가 해당 컬럼 무시)

**재현 시나리오**:
1. 엑셀 템플릿에 `난이도: 상` 컬럼 추가 후 업로드 → 파싱 무시
2. QuizCreate에서 은행별 출제 수/난이도 배점 설정 UI 없음
3. 문항 목록에 난이도 배지 없음

**영향도**: 경북대 고객사 핵심 요구사항 미충족. 프로토타입 시연 시 치명적 인상 저하. 단순 UI 추가로 해결 불가 — 데이터 모델 재설계 선행 필요
**수정 권고안**: 1단계로 Question 모델에 `difficulty: 'easy'|'medium'|'hard'|null`, `groupTag: string|null` 필드 추가 및 mockData 전수 갱신 → 2단계로 엑셀 파싱 컬럼 확장 → 3단계로 QuizCreate UI 추가

---

### CR-04. URL 직접 접근 시 권한 체크 없음

**관련 PM**: PM2, PM3, PM4 (3개 PM 공통 지적)
**재현 시나리오**: 브라우저 주소창에 `/quiz/1/grade` 직접 입력 → 로그인 없이 채점 화면 진입 가능. `/quiz/1/stats`도 동일
**영향도**: 학생이 퀴즈 ID를 알면 채점 대시보드 접근 가능. 전체 학생 점수 및 채점 내역 열람 가능
**수정 권고안**: GradingDashboard, QuizCreate, QuizEdit, QuestionBank, QuestionBankList, QuizStats 전체에 ProtectedRoute 컴포넌트 적용. 현재 QuizAttempt에만 role 체크 구현되어 있음

---

### CR-05. 정답 데이터 클라이언트 번들 노출

**관련 PM**: PM3, PM4
**재현 시나리오**: 브라우저 개발자도구 → Sources → `mockData-RgG_sBRO.js` 검색 → `AUTO_CORRECT_ANSWERS` 객체에서 전체 정답 열람 가능
**영향도**: 시험 시행 중 학생이 실시간으로 정답 확인 가능. 시험 공정성 훼손
**수정 권고안**: 프로토타입 단계에서 `/* 개발용 데이터 — 실운영 제거 필요 */` 주석 명시. 실운영 전환 시 정답 검증은 반드시 서버사이드에서만 수행

### CR-06. mockQuizzes 배열 직접 변이 (React 불변성 위반)

**관련 PM**: PM3
**재현 시나리오**: 퀴즈 생성 후 새로고침 → 생성한 퀴즈 소멸. `mockQuizzes.push()`가 메모리에만 추가되고 localStorage 미동기화
**영향도**: 교수자가 퀴즈를 생성해도 새로고침 시 사라짐. 프로토타입 시연 중 신뢰도 하락
**수정 권고안**: QuestionBankContext 패턴과 동일하게 QuizContext 또는 useState 기반 관리로 전환. localStorage 동기화 추가

---

### CR-07. 응시 횟수 초과 차단 로직 없음

**관련 PM**: PM2, PM4
**재현 시나리오**: allowAttempts=1로 설정된 퀴즈에서 학생이 2회, 3회 반복 응시 → 모두 허용됨
**영향도**: 시험 정책 전체 무력화. 프로토타입 시연 시 가장 먼저 발견될 수 있는 치명적 버그
**수정 권고안**: QuizAttempt 진입 시 `getStudentAttempts()`로 해당 학생의 응시 횟수 확인 → `allowAttempts` 초과 시 응시 차단 화면 표시

## 6. PM 간 의견 충돌 및 조율 사항

### 충돌 사항 없음
PM1~PM5 간 이슈 방향이 대체로 일치한다. 다만 우선순위 관점에서 조율이 필요한 지점 2개를 명시한다.

**[조율 1] 정답 클라이언트 노출 대응 시점**
- PM3: 프로토타입 단계에서도 완화 가능 (정답 해시 비교 방식)
- PM5: Out-of-scope로 분류 (서버 전환 후 해결)
- **Leader 판단**: 고객사 시연 대상에 실제 학생이 포함될 경우 즉시 주석 경고 및 내부 관계자 전용 시연 프로세스 수립 필요. 완화 코드 작업보다 시연 범위 확인이 선행되어야 함

**[조율 2] GradingDashboard 컴포넌트 분리 시점**
- PM3: 즉시 분리 권고 (단일 파일 39 kB, 상태 변수 과다)
- PM5: MVP2 범위 외 언급 없음
- **Leader 판단**: 현 스프린트에서는 분리보다 CR-04(ProtectedRoute) 및 mockQuizzes 전환이 우선. GradingDashboard 분리는 MVP2 기능 추가 시 자연스럽게 병행
