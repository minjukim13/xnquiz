## 1. 핵심 사용자 시나리오 (Happy Path) 정합성 검토

### 1-1. 시나리오별 현황

| **시나리오** | **대상 사용자** | **현재 구현** | **누락/문제** | **판정** |
|---|---|---|---|---|
| 퀴즈 생성 → 발행 | 교수자 | 기본정보 입력 + 문항 구성 2탭, 발행 버튼 구현 | 임시저장 버튼 존재하나 실제 저장 로직 없음 (UI만) | 부분 구현 |
| 문제은행 문항 추가 | 교수자 | 은행 선택 → 문항 선택 2단계 모달 구현 | 랜덤 출제 설정 없음, 복수 은행 동시 선택 불가 | 부분 구현 |
| 직접 문항 추가 | 교수자 | AddQuestionModal로 유형 선택 후 추가 | 정답/선택지 입력 UI 별도 확인 필요 | 부분 구현 |
| 퀴즈 응시 (학생) | 학습자 | 타이머, 문항 렌더링, 제출 완료 모달 구현 | 복수 응시 시 이전 응시 결과 비교 불가, 재응시 버튼 없음 | 부분 구현 |
| 채점 (교수자) | 교수자 | 문항중심/학생중심 2모드, 수동채점 인터페이스 구현 | 채점 완료 후 성적 확정/공개 분리 없음 | 부분 구현 |
| 결과 확인 (학생) | 학습자 | 자동채점 즉시 결과, 수동채점 대기 안내 | 최종 성적 확정 후 학생 통보 정책 없음 | 부분 구현 |
| 퀴즈 통계 확인 | 교수자 | QuizStats 페이지 존재 (별도 파일) | 문항별 정답률, 오답 분포 분석 구현 여부 추가 확인 필요 | 확인 필요 |
| 문제은행 관리 | 교수자 | CRUD, 검색/필터, Excel 업로드, 은행 간 복사 구현 | 난이도/그룹 메타데이터 없음 | 부분 구현 |
| 주차별 퀴즈 목록 | 교수자/학습자 | week/session 필드는 있으나 주차별 필터 뷰 없음 | 주차 기준 그룹핑/정렬 UI 없음 | 미구현 |

### 1-2. 퀴즈 상태 전이 정합성

현재 정의된 퀴즈 상태: `draft` → `open` → `grading` → `closed`

```
[초안(draft)] --발행--> [진행중(open)] --마감시점--> [채점중(grading)] --채점완료--> [종료(closed)]
```

**문제점:**
- `open` → `grading` 전환 트리거가 명확하지 않음 (마감 일시 자동 전환인지, 교수자 수동 전환인지 불명확)
- `grading` → `closed` 전환 조건 불명확 (채점 100% 완료 시 자동인지, 교수자 수동 확정인지)
- 성적 공개(Published) 상태가 별도로 존재하지 않음 — 채점 완료 = 즉시 성적 공개인지 정책 미정의
- `draft` 상태에서 편집 후 재발행 시나리오 처리 누락

## 4. 기능 정책 완결성 검토

### 4-1. 재응시 정책

**현재 상태:** QuizCreate에서 allowAttempts(1/2/3/무제한) + scorePolicy(최고점수/최신점수/평균) 설정 가능. getStudentAttempts()에서 정책별 점수 반환 로직 구현됨.

**미정의 항목:**
- 응시 횟수 초과 시 학생 화면 처리 (현재 미구현, 재응시 버튼 없음)
- 복수 응시 간 냉각 시간(Cooldown) 정책 없음
- 이전 응시 답안 학생에게 공개 여부 정책 없음
- 재응시 시작 시점에 이전 답안 초기화 여부 정책 없음

### 4-2. 지각 제출 정책

**현재 상태:** allowLateSubmit 체크박스 존재, 활성화 여부만 설정 가능.

**미정의 항목:**
- 지각 허용 시간 범위 (마감 후 몇 분/시간까지)
- 지각 제출 시 감점 정책 (없음/고정 감점/시간비례 감점)
- 지각 제출 학생 채점 우선순위 처리

### 4-3. 점수 공개 정책

**현재 상태:** showAnswerAfter 토글로 "제출 후 정답 공개" 설정 가능. 자동채점 결과는 제출 즉시 ResultModal로 표시.

**미정의 항목:**
- 교수자가 성적을 일괄 확정 후 공개하는 별도 "성적 공개(Release)" 단계 없음
- 수동채점 완료 후 학생 통보(알림) 정책 없음
- 전체 채점 완료 전 자동채점 결과만 먼저 공개할지 여부 정책 없음
- 채점 기간 중 점수 임시 비공개 정책 없음

### 4-4. 채점 방식 정책

**현재 상태:** 자동채점(완전 일치), 수동채점 분리. 부분점수 없음.

**미정의 항목:**
- 복수선택(multiple_answers) 문항의 부분 점수 정책 (현재 자동채점 true이나 부분점수 없음)
- 서술형 채점 기준표(루브릭) 입력/활용 기능 없음
- 동점자 처리 정책 없음 (통계/순위에서)

| 퀴즈 응시 | 불가 (redirect) | 가능 | 미정의 |
| 채점 대시보드 | 접근 가능 | URL 직접 접근 시 미차단 | 미정의 |
| 문제은행 관리 | 접근 가능 | 접근 불가 (링크 없음, URL 직접 접근 미차단) | 미정의 |
| 통계/결과 | 접근 가능 | 접근 불가 (링크 없음) | 미정의 |

**Critical 이슈:** URL 직접 접근 시 role guard가 일부 페이지에만 적용됨. QuizAttempt에만 role !== 'student' 리다이렉트 구현됨.

### 5-2. 개선 필요 사항

- 모든 교수자 전용 라우트에 ProtectedRoute 컴포넌트 적용 필요
- 학생 전용 라우트에도 동일하게 적용 필요
- 조교(TA) 역할 신설 시 채점 허용/퀴즈 생성 제한 등 권한 매트릭스 정의 필요

## 8. 정책 개선안 요약

### 8-1. 즉시 확정 필요 정책 (Critical)

**[P1] 사용 중인 문제은행 삭제 정책**
- 현재: deleteBank 시 즉시 삭제 + 소속 문항 전체 삭제
- 개선안: 퀴즈에서 사용 중인 은행은 삭제 불가 차단. 미사용 은행만 삭제 허용. 차단 시 "현재 N개 퀴즈에서 사용 중입니다" 안내

**[P2] 응시 횟수 초과 차단 정책**
- 현재: 응시 횟수 제한 설정은 가능하나 실제 차단 로직 없음
- 개선안: QuizAttempt 진입 시 getStudentAttempts로 해당 학생의 응시 횟수 확인 → allowAttempts 초과 시 응시 차단 및 안내 화면 표시

**[P3] 퀴즈 상태 전이 자동화 정책**
- 현재: draft → open은 교수자 수동, 이후 전이는 자동인지 수동인지 불명확
- 개선안: open → grading은 마감 일시 경과 시 자동 전환, grading → closed는 교수자 "성적 확정" 버튼으로 수동 전환

**[P4] 교수자 전용 페이지 role guard 전체 적용**
- 현재: QuizAttempt만 role 체크
- 개선안: GradingDashboard, QuizCreate, QuizEdit, QuestionBank, QuestionBankList 모든 페이지에 role guard 적용

### 8-2. MVP2 개발 전 확정 필요 정책

**[P5] 지각 제출 세부 정책**
- 허용 시간 범위 옵션: 마감 후 N분, N시간, 당일 자정까지 (선택)
- 감점 정책 옵션: 없음 / 고정 감점(N점) / 시간비례 감점 중 교수자 선택

**[P6] 복수선택 문항 부분 점수 정책**
- 옵션 A: 전체 정답 선택 시에만 만점, 오답 하나라도 포함 시 0점
- 옵션 B: 정답 개수 기준 비율 배점 (예: 정답 3개 중 2개 = 2/3점)
- 옵션 C: 교수자가 문항 생성 시 선택

**[P7] 성적 공개 단계 정의**
- 채점 완료와 성적 공개를 분리하여 "성적 확정 및 공개" 버튼 도입
- 공개 전 학생에게는 "채점 중" 상태 유지

## 10. 자체 리뷰

**잘 구현된 영역:**
- 교수자/학생 뷰 분리, 채점 2모드(문항중심/학생중심), Excel 업로드, 자동채점 로직이 프로토타입 수준에서 비교적 탄탄하게 구현됨
- 점수 정책(최고/최신/평균) 구현은 경쟁 LMS 대비 손색없음

**핵심 리스크:**
1. **응시 횟수 차단 미구현**: 설정은 되어 있으나 실제 차단이 없어 무제한 응시가 가능함. 프로토타입 시연 시 혼선 발생 가능
2. **사용 중 문제은행 삭제 시 데이터 손실**: Critical 버그 수준
3. **URL 직접 접근 권한 미체크**: 학생이 채점 대시보드 URL을 직접 입력하면 접근 가능한 상태
4. **고객사 핵심 요구(경북대: 난이도/그룹 기반 랜덤 출제)**: 데이터 모델부터 재설계 필요, 단순 UI 추가로 해결 불가
5. **임시저장 버튼 UI만 존재**: 클릭 시 아무 동작 없음 — 학생/교수자 혼란 유발 가능
