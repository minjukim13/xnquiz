## 작성 목적

xnquiz 가 대체할 LearningX(Canvas + custom.js) 화면 대비, **어떤 UI/UX 문제를 어떻게 개선했는지** 화면 단위로 명세한다. 백엔드 결합 진행을 위한 권한 정책, 상태 정의, 컴포넌트 동작, 버튼별 액션, Edge Case 를 함께 정리하며, 후속 API 요청서(ASD)의 입력 자료가 된다.

## 공통 사용자 역할 정의

| **코드** | **역할** | **설명** |
|---|---|---|
| **INSTRUCTOR** | 교수자 | 퀴즈 생성/편집/채점, 본인 과목 데이터 전체 권한 |
| **TA** | 조교 | 본 프로토타입 범위 외 (교수자 권한 일부 위임 예정) |
| **STUDENT** | 학습자 | 본인에게 공개된 퀴즈 응시, 본인 응시 결과 조회 |
| **ADMIN** | 운영자 | 교수자 권한 전부 포함, 추가로 시스템/학교 단위 운영 권한 |

> 권한 분기 규칙: 교수자 전용 기능은 `(INSTRUCTOR || ADMIN)` 조건으로 처리한다. ADMIN 단독 분기는 사용하지 않는다.

## 공통 상태 정의

### 퀴즈 상태 (quiz.status)

| **상태값** | **표시 라벨** | **의미** |
|---|---|---|
| `draft` | 임시저장 | 교수자만 보임, 학생 화면에 노출되지 않음 (자동 비공개) |
| `open` | 진행중 | 학생 응시 가능 상태 |
| `grading` | 채점중 | 마감 후 수동채점 진행 중 (학생에게는 "마감" 으로 표시) |
| `closed` | 마감 | 응시 종료 (단, 지각 제출 허용 시 lateSubmitDeadline 까지 진행중 유지) |

### 화면용 파생 상태 (resolveDisplayStatus)

| **파생 상태** | **조건** | **표시 라벨** |
|---|---|---|
| `scheduled` | `status=open` && 현재 시각 < `startDate` | 예정 |
| `closed` (자동) | `status=open` && 현재 시각 > `dueDate` (지각 제출 비활성) | 마감 |
| 원래 status 유지 | 위 조건 외 | 위 표 그대로 |

### 학생 공개 상태 (quiz.visible)

| **visible** | **status** | **학생 화면 노출** |
|---|---|---|
| `true` | `open` / `grading` / `closed` | 노출 |
| `false` | 모든 상태 | 비노출 (숨김) |
| 무관 | `draft` | 자동 비노출 (visible 값과 무관) |

### 응시 상태 (학생 본인 기준)

| **상태** | **의미** |
|---|---|
| 미응시 | attempt 가 없음 |
| 응시중 | attempt 생성됨, submitted=false |
| 응시완료 | submitted=true |
| 채점 대기 | 수동채점 문항이 있고 manualPending > 0 |

## 공통 UI 규칙

1. **디자인 시스템**: 현재 디자인 시스템이 별도 구축 중이며, 본 화면들의 UI 스타일(색상, 타이포, 컴포넌트 디테일)은 디자인 시스템 확정 후 일괄 반영한다. 본 URD 는 컴포넌트 구조와 동작 명세에 집중하며, 시각 스타일 상세는 디자인 시스템에 따른다.
2. **버튼**: 액션 위계(기본/보조/위험/최소)별로 컴포넌트화된 버튼을 사용하며, 화면 단위 색상 하드코딩은 금지한다. 최종 스타일은 디자인 시스템 확정에 따른다.
3. **모달**: 타이틀/설명/푸터 구조는 공통 컴포넌트를 사용하며, 인라인 폰트/패딩 오버라이드는 금지한다.
4. **로딩**: api 모드는 Skeleton 표시, mock 모드는 즉시 데이터 표시
5. **빈 상태**: 아이콘 + 안내 문구 형식 통일
6. **Toast**: 우측 상단, 4초 자동 해제
7. **확인 모달 (ConfirmDialog)**: 작성 데이터 손실 가능 화면에서만 사용, 단순 조회/필터 변경에는 사용 금지
8. **권한 미충족 시**: 홈으로 리디렉션 처리
