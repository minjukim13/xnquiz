XN Quiz 구현 현황 보고 — 발표 대본

슬라이드별 핵심 메시지를 격식체 1~3문장으로 정리. 기능 시연 슬라이드는 직접 시연으로 대체하므로 대본에서 제외.

1. 표지

XN Quiz UX 프로토타입 — 구현 현황 보고

오늘은 XN Quiz 프로토타입의 현재까지의 구현 현황을 보고드리겠습니다. Canvas LMS의 Quiz 기능을 재구현하면서 고객사 요구사항을 반영한 결과물이며, 디자인 시스템은 별도로 구축을 진행 중입니다.

2. Overview · 보고 핵심 메시지

XN Quiz 프로토타입 — 무엇을 만들었고, 어디까지 가능한가

오늘 보고드릴 내용은 네 가지입니다. 프로젝트 진행 방식, Canvas 대비 UI/UX 개선 포인트, 디자인 시스템 진행 상태, 그리고 로드맵입니다.

핵심은 단순합니다. Canvas Quiz 기본 기능을 재구현하면서 고객사 요구사항 18개를 전 항목 반영했고, 지금부터는 사용성과 일관성을 견고하게 다듬는 단계로 진입했다는 것입니다.

3. Process · AI 기반 개발 프로세스

Claude Code 기반 병렬 개발 플로우

기존에는 기획서·정적 시안과 같은 정적인 산출물로 의사결정을 받아왔지만, 이번에는 AI 에이전트와의 협업을 통해 실제 동작이 검증 가능한 동적 산출물(프로토타입) 까지 만들어냈다는 점이 가장 큰 변화입니다. 글이나 도식만으로는 잡히지 않던 사용 흐름의 허점을 직접 화면에서 확인하고 다듬을 수 있었습니다.

4. Process · AI 에이전트 운영

다중 PM 검토 에이전트를 활용한 산출물 동시 검증

기존의 순차 프로세스는 기획 → 디자인 → 개발 → QA → PO 단계가 끝나야 다음이 시작되어 단계 간 대기 시간이 길고, 후반에 발견된 이슈는 앞 단계 재작업으로 이어졌습니다. 다중 에이전트 병렬 운영은 이 흐름을 동시 진행으로 바꿔, 두 가지 큰 이점을 만듭니다.

첫째, 처리 속도와 효율성의 극대화입니다. 거대하고 복잡한 작업을 여러 에이전트가 나누어 동시에(병렬로) 처리하므로 전체 소요 시간이 획기적으로 단축되고, 각 에이전트가 가용 자원을 최대한 활용해 단계별 병목을 줄입니다.

둘째, 복잡한 문제의 단순화입니다. 큰 문제를 작고 관리하기 쉬운 하위 문제로 모듈화하고, 각 에이전트가 특정 영역(디자인/기획/개발/QA/PO)에 특화된 전문가 형태로 협력해 문제를 더 효율적으로 해결합니다.

5. UX · Canvas → xnquiz 개선 종합

Canvas LMS의 UX 한계, 5대 영역에서 해소

학교 요구사항과 Canvas Baseline을 함께 분석한 결과, 개별 기능 단위가 아니라 사용 흐름 전반의 사용성 문제라는 결론에 도달했습니다. 다음 슬라이드부터는 5대 개선 축을 화면 단위 Before/After로 비교해 보여드리겠습니다.

6. UX 비교 1 · 문제 유형 선택

라벨만 보고 추측에서 예시 문항으로 직관적 선택

Canvas는 문제 유형을 라벨 텍스트만으로 제공해, 출제자가 의미를 추측해야 했습니다. XN Quiz는 카드형 UI에 호버 시 예시 문항이 미리 보이도록 해, 유형 선택 단계에서의 인지 부하를 줄였습니다.

7. UX 비교 2 · 자산 재활용

매번 새로 만들던 방식에서 문제은행·퀴즈 가져오기로

Canvas는 다른 과목의 문제은행이나 시험을 가져다 쓰기 어려워, 동일한 시험도 매번 처음부터 다시 구성해야 했습니다. XN Quiz는 문제은행 가져오기/내보내기와 퀴즈 가져오기를 지원해, 한 번 만들어둔 자산을 여러 과목에서 그대로 재활용할 수 있도록 했습니다.

8. UX 비교 3 · 시험 목록

통계 화면까지 들어가야 보였던 정보, 목록에서 한눈에

Canvas는 응시율이나 평균 점수 같은 수치를 보려면 퀴즈 목록 → 퀴즈 상세 → 퀴즈 통계 화면까지 단계별로 들어가야 했습니다. XN Quiz는 목록 단계에서 응시율, 평균 점수, 채점 진행률 같은 핵심 수치를 바로 노출해, 별도 화면 진입 없이 한눈에 확인할 수 있도록 했습니다.

9. UX 비교 4 · 채점 방식

학생별 채점만 가능했던 방식에서 학생별 + 문항별 양방향 채점으로

Canvas는 학생별 채점만 지원해, 수동 채점이 필요한 문항을 한눈에 모아서 보지 못하고 학생을 한 명 한 명 넘겨가며 채점해야 했습니다. XN Quiz는 학생별 채점에 더해 문항별 일괄 채점을 추가로 지원해, 같은 문항의 모든 응답을 한 화면에서 비교하면서 일관된 기준으로 채점할 수 있도록 했습니다.

10. UX 비교 5 · 재채점 흐름

재채점 지원 유형 확대 + 진입 경로 단일화

Canvas는 재채점 가능한 유형이 제한적이고 진입 경로도 분산되어 있었습니다. XN Quiz는 지원 유형을 확장하고 진입 경로를 한 곳으로 통일해, 채점 정책 변경 시 발생하는 운영 부담을 줄였습니다.

11. UX 비교 6 · 통계 / 분석

Canvas는 표 형태 수치 중심, XN Quiz는 차트 기반 시각화

Canvas의 통계 화면은 점수와 지표가 표 형태의 숫자 나열로 제공되어, 한눈에 전체 양상을 파악하거나 의사결정에 바로 쓰기 어렵습니다. XN Quiz는 차트 기반 시각화로 재구성해, 점수 분포·문항 변별도·정답률을 직관적으로 확인할 수 있도록 했습니다.

12. DS · 디자인 시스템

여러 프로젝트가 한 LMS 안에 들어가는 만큼, 시각 언어 통일이 필요

XN Quiz 한 프로덕트가 아니라, 같은 LMS 안에 여러 모듈이 함께 들어가는 구조입니다. 따라서 시맨틱 컬러 토큰과 공용 컴포넌트 체계를 먼저 정립해, 새로운 화면이 추가될 때마다 일관성이 자동으로 보장되도록 했습니다.

(시연 구간)

핵심 기능은 프로토타입 직접 시연으로 대체 — Reference 슬라이드의 항목들을 화면에서 차례로 보여드림.

13. Scope · 프로토타입 vs 백엔드

지금 시연 가능한 것 / 백엔드 팀이 실제로 구현해야 할 것

프로토타입의 목적은 사용 흐름과 화면을 시연하는 것입니다. 좌측은 지금 시연으로 보여드릴 수 있는 영역, 우측은 실제 운영을 위해 서버에서 검증·강제·기록되어야 하는 백엔드 영역입니다.

이 두 영역을 분리해 노출해 둔 이유는, 개발팀 인계 시 무엇이 시연용이고 무엇이 실구현 대상인지 혼선을 줄이기 위함입니다.

14. Roadmap · 현재 포커스

백엔드 신규 구현은 일시 중단 — UI/UX 사용성과 디자인 시스템 안정화에 집중

백엔드는 직접 시도해 실 동작까지 확인했지만, 보안·IP 검증 등 운영 신뢰성이 핵심인 영역은 실제 개발을 잘 아는 개발자가 진행하는 것이 맞다고 판단해 중단했습니다. 현 단계의 우선순위와 본인의 책임 영역은 UI/UX와 디자인 시스템을 견고하게 다듬는 일이며, 그 단계가 마무리된 이후 백엔드 본구현을 이어가는 것이 적절하다고 봅니다.

15. Roadmap · 장기 방향

1차 목표 너머 — 다음 단계로 검토할 영역

이번 1차에서는 기존 Canvas Quiz 기능과 고객사 요구사항 적용으로 의도적으로 범위를 좁혔습니다. 1차 표준이 안정화된 이후에는 Canvas New Quizzes의 신규 기능, 추가 문항 유형, 그리고 고도화 영역을 단계적으로 검토해 확장해 나가고자 합니다.

16. Retrospective · AI 활용 회고

자연어 지시에서 정형화된 AI 사용법으로

가장 크게 느낀 학습 포인트는 AI에게 무엇을 시키는지보다 어떻게 시키는지가 결과 품질을 결정한다는 점입니다. 이번 1차 진행에서는 주로 자연어 지시 기반으로 구현을 진행했고, CLAUDE.md·Skill·Hook 같은 정형화된 도구도 일부 활용해 보았으나 완전히 그 방식으로 정착시키진 못했습니다.

같은 입력에 같은 형식의 산출물이 나오는 재현 가능한 작업 흐름은 다음 프로젝트에서 본격적으로 적용해 보고자 하는 다음 단계 과제이며, 자주 쓰는 산출물을 표준 자산으로 등록해 개인 작업이 회사 자산으로 인계되도록 운영하는 방향을 함께 검토해 보겠습니다.
