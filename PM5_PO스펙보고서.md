# PM5 — PO 스펙 정의서 (XN Quizzes Prototype)

**작성일**: 2026-03-25
**작성자**: PM5 (시니어 PO)
**참조**: PM1 디자인검토보고서, PM2 기획검토보고서, PM3 개발검토보고서, PM4 QA테스트보고서
**포맷**: Jira Confluence 업로드용

---

## 0. Executive Summary

XN Quizzes 프로토타입은 교수자 관점의 퀴즈 생성-편집-채점-통계 확인 핵심 플로우를 시각적으로 검증하는 수준에 도달해 있다. 전체 5개 페이지(QuizList, QuizCreate, QuizEdit, GradingDashboard, QuizStats)와 문제은행(QuestionBank)이 구현되어 UI 구조는 완성도 있게 갖춰졌다. 그러나 채점 저장, 퀴즈 생성 실제 반영, 문제은행 CRUD 등 핵심 인터랙션이 로컬 상태에만 머물러 있어 Happy Path가 단절된 상태다. MVP1 완료 기준으로는 이 단절 지점 3개(저장 로직 연결, URL 파라미터 연동, 발행 후 목록 반영)가 최우선 수정 대상이다. 팀 규모(2-3인)와 프로토타입 단계를 고려할 때, 본 문서에 정의된 Must-have 항목에 집중하고 나머지는 MVP2 이후로 명확히 구분하여 개발 리소스를 보호하는 것이 현실적인 전략이다.

## 1. 기능별 Acceptance Criteria

### 1.1 퀴즈 생성

**AC-01: 기본 정보 유효성 검증**
- Given: 교수자가 퀴즈 생성 화면에서 정보를 입력할 때
- When: 제목이 비어있거나, 시작일이 없거나, 마감일이 없거나, 시작일 >= 마감일인 경우
- Then: "다음: 문항 구성" 버튼이 비활성화 상태를 유지하고, 역순 날짜의 경우 "마감일은 시작일보다 늦어야 합니다" 인라인 오류 메시지가 표시된다.

**AC-02: 문항 수 검증 후 발행**
- Given: 교수자가 문항 구성 탭에 있을 때
- When: 문항이 0개인 상태에서 "발행하기" 버튼을 클릭하면
- Then: 버튼이 비활성화 상태로 유지되고 "문항을 1개 이상 추가해야 발행할 수 있습니다" 안내 문구가 표시된다.

**AC-03: 총점 0점 발행 차단**
- Given: 문항이 존재하지만 모든 문항의 배점이 0점인 경우
- When: 교수자가 "발행하기"를 클릭하면
- Then: 발행이 차단되고 "총 배점이 0점인 퀴즈는 발행할 수 없습니다" 메시지가 표시된다.

**AC-04: 발행 후 목록 반영**
- Given: 교수자가 유효한 퀴즈 정보와 문항을 입력한 후
- When: "발행하기" 버튼을 클릭하면
- Then: 퀴즈가 프로토타입 상태 저장소(mockData 또는 전역 state)에 추가되고, QuizList 화면으로 이동 시 신규 퀴즈가 목록에 노출되며 status는 'open'으로 표시된다.

**AC-05: 임시저장**
- Given: 교수자가 퀴즈 생성 중 임시저장 버튼을 클릭하면
- When: 현재 입력 내용(제목, 날짜, 문항 목록)이 저장되면
- Then: "임시저장 완료" 토스트 메시지가 1.5초간 표시되고, 이후 해당 퀴즈는 QuizList에서 status='draft'로 노출된다.

---

### 1.2 퀴즈 편집

**AC-06: URL 파라미터 기반 데이터 로딩**
- Given: 교수자가 QuizList에서 특정 퀴즈의 "편집" 버튼을 클릭하면
- When: `/quiz/:id/edit` 페이지가 로드될 때
- Then: URL의 `:id`에 해당하는 퀴즈의 실제 데이터(제목, 날짜, 문항 목록)가 폼에 자동으로 채워진다. 존재하지 않는 id인 경우 "퀴즈를 찾을 수 없습니다" 상태와 함께 목록으로 돌아가는 버튼이 표시된다.

**AC-07: 편집 저장 후 목록 반영**
- Given: 교수자가 발행된 퀴즈를 편집하고 저장하면
- When: "변경사항 저장" 버튼을 클릭하면
- Then: 변경된 내용이 저장소에 반영되고 QuizList에서 수정된 제목과 설정이 확인된다.

**AC-08: 발행된 퀴즈 편집 경고**
- Given: status='open' 또는 'grading' 상태의 퀴즈를 편집할 때
- When: 교수자가 문항 내용을 수정하거나 삭제하려 하면
- Then: "이미 발행된 퀴즈입니다. 현재 응시 중인 학생이 있을 수 있습니다. 문항을 수정하시겠습니까?" 확인 모달이 표시된다. 확인 선택 시에만 수정이 진행된다.

---

### 1.3 채점 대시보드

**AC-09: 채점 저장 — 문항 중심 모드**
- Given: 교수자가 문항 중심 모드에서 특정 학생의 점수를 입력하고
- When: "저장" 버튼을 클릭하면
- Then: 해당 학생의 점수가 저장소에 반영되고, 저장 버튼 영역에 "저장됨" 상태 표시가 나타난다. 페이지를 새로고침해도 저장된 점수가 유지된다.

**AC-10: 채점 저장 — 학생 중심 모드**
- Given: 교수자가 학생 중심 모드에서 특정 학생의 수동채점 문항 점수를 입력하고
- When: "채점 저장" 버튼을 클릭하면
- Then: 해당 학생의 전체 채점 결과가 저장되고, 학생 목록에서 해당 학생의 상태가 "미채점 → 채점 완료"로 전환된다.

**AC-11: 점수 입력 범위 제한**
- Given: 교수자가 점수 입력란에 값을 입력할 때
- When: 0 미만이거나 해당 문항 배점 초과이거나 소수점 값을 입력하면
- Then: 저장 버튼이 비활성화 상태를 유지하고, 범위 초과 시 인라인 오류 표시가 나타난다. 저장 가능한 값은 0 이상, 해당 문항 배점 이하의 정수만 허용된다.

**AC-12: 이탈 전 경고**
- Given: 교수자가 점수를 입력했으나 저장하지 않은 상태에서
- When: 다른 페이지로 이동하거나 브라우저 탭을 닫으려 하면
- Then: "저장되지 않은 채점 내용이 있습니다. 페이지를 이탈하시겠습니까?" 확인 모달이 표시된다.

**AC-13: short_answer 채점 방식**
- Given: 단답형(short_answer) 문항이 채점 대시보드에 표시될 때
- When: 교수자가 문항 유형 필터를 확인하면
- Then: 단답형 문항은 "수동 채점 필요" 탭에 포함되어 표시된다. (자동채점 제외, 수동채점 대상으로 처리)

---

### 1.4 퀴즈 통계

**AC-14: 채점 미완료 상태 표시**
- Given: 일부 학생에 대한 채점이 완료되지 않은 상태에서 통계 페이지에 접근하면
- When: 페이지가 로드될 때
- Then: 상단에 "채점 미완료: N명 미채점 — 현재 통계는 채점 완료된 M명 기준입니다" 안내 배너가 노출된다.

**AC-15: 응시자 0명 Empty State**
- Given: 아직 제출자가 0명인 퀴즈의 통계 페이지에 접근하면
- When: 페이지가 로드될 때
- Then: 차트와 점수 분포 대신 "아직 제출한 학생이 없습니다" Empty State 화면이 표시된다.

---

### 1.5 문제은행

**AC-16: 문항 추가 저장 동작**
- Given: 교수자가 문제은행에서 "문항 추가" 버튼을 클릭하고 내용을 입력한 후
- When: 모달의 "추가" 버튼을 클릭하면
- Then: 새 문항이 문제은행 목록에 즉시 반영되어 표시된다.

**AC-17: 문항 편집 저장 동작**
- Given: 교수자가 문항의 편집 아이콘을 클릭하여 인라인 편집 폼을 열고 수정한 후
- When: "저장" 버튼을 클릭하면
- Then: 수정된 내용이 목록에 즉시 반영되고 편집 폼이 닫힌다.

**AC-18: 문항 삭제 동작**
- Given: 교수자가 문항의 삭제 아이콘(휴지통)을 클릭하면
- When: 삭제 확인 모달에서 "삭제" 버튼을 클릭하면
- Then: 해당 문항이 목록에서 제거된다. 사용 중인 퀴즈가 있는 경우(usageCount > 0) "N개 퀴즈에서 사용 중입니다. 삭제하시겠습니까?" 경고 문구가 확인 모달에 포함된다.

---

### 1.6 재채점

**AC-19: 재채점 범위 제한**
- Given: 채점이 완료된 퀴즈에서 교수자가 재채점을 실행하면
- When: "재채점 실행" 버튼을 클릭하면
- Then: 자동채점(autoGrade=true) 문항의 점수만 재산출된다. 수동으로 입력된 점수(essay, short_answer, file_upload 등 수동채점 문항)는 변경되지 않고 유지된다. 재채점 완료 후 영향받은 학생 수와 점수 변동 요약이 표시된다.

**AC-20: 재채점 후 통계 갱신**
- Given: 재채점이 완료된 후
- When: 교수자가 통계 페이지를 확인하면
- Then: 점수 분포, 평균, 최고/최저 점수가 재채점 결과 기준으로 갱신된 값을 표시한다.

## 2. 정책 결정 사항 (Policy Decisions)

> 아래는 PM1~PM4 검토에서 확인된 정책 공백에 대한 PO 최종 결정 사항이다. 근거 없이 "미정"으로 남겨두는 것은 개발 착수를 지연시키므로 현 시점에서 확정한다.

---

### 2-1. 채점 저장 방식

**결정**: 수동저장 + "저장" 버튼 방식 채택 (자동저장 미채택)

**이유**:
- 자동저장은 입력 도중 중간값(예: 빈 값, 일부만 입력된 상태)이 저장될 위험이 있어 채점 정확성을 해칠 수 있다.
- 교수자가 저장 여부를 명확히 인지하는 수동저장이 채점 행위의 신뢰성을 높인다.
- 단, 미저장 상태에서 페이지 이탈 시 경고 모달(AC-12)은 필수 구현한다.

**프로토타입 구현 방식**: localStorage에 채점 결과를 persist하여 새로고침 후에도 유지. 실데이터 전환 시 PATCH API 호출로 교체.

---

### 2-2. 퀴즈 상태 전이 규칙 (published/status 필드 충돌 해결)

**결정**: `status` 필드를 단일 진실 공급원(SSOT)으로 사용. `published` 필드 폐기.

**상태 정의**:
- `draft`: 미발행 초안. 학생 접근 불가. 편집 자유.
- `open`: 발행 완료. 응시 가능 기간. 문항 수정 시 경고 표시.
- `grading`: 응시 기간 종료 후 채점 진행 중. 수동채점 문항이 없는 경우 마감 즉시 `closed`로 전이.
- `closed`: 채점 완료. 결과 열람 가능.

**상태 전이 트리거**:
- `draft → open`: "발행하기" 버튼 클릭 (발행 전 체크리스트 통과 조건 필수)
- `open → grading`: 마감 일시 도달 시 자동 전이 (MVP1에서는 mockData 수동 변경으로 시뮬레이션)
- `grading → closed`: 전체 채점 완료 시 자동 전이 또는 교수자 수동 종료 버튼 (MVP2에서 구현)

**이유**: 두 필드가 독립적으로 존재하면 `published=true, status='draft'` 같은 논리적 모순 상태가 발생 가능하다(PM2 C-03, PM4 F-02 근거). 단일 필드로 통일하면 상태 관리 로직이 단순해지고 버그 발생 면적이 줄어든다.

---

### 2-3. 지각 제출 정책

**결정 (MVP1)**: 지각 제출 허용 토글은 유지하되 세부 정책(허용 기간, 감점 정책)은 MVP2로 이관. MVP1에서는 토글 ON 시 "마감 후 제출 허용 (감점 없음, 기간 제한 없음)" 고정 동작.

**결정 (MVP2)**: 아래 세부 옵션 추가 구현
- 허용 기간: 마감 후 N일 이내 입력 필드 (기본값: 7일)
- 감점 정책: 없음 / 일 단위 N% 감점 중 선택 (기본값: 없음)
- 채점 대시보드에서 지각 제출 학생에게 "Late" 배지 표시

**이유**: 현재 토글만 있고 세부 동작이 미정인 상태는 프로토타입 데모에서 혼란을 준다. MVP1에서는 허용/불허 이분법으로 단순화하고, 세부 정책은 실제 교수자 피드백 수집 후 MVP2에서 정교화한다.

---

### 2-4. 정답 공개 시점

**결정**: 4가지 옵션 중 선택 방식으로 구현 (MVP2에서 세분화)

**MVP1 기준**: "제출 후 즉시 공개" 단일 옵션으로 단순화.

**MVP2 기준 옵션**:
1. 제출 직후 즉시 공개
2. 퀴즈 마감 후 공개 (기본 권장값)
3. 채점 완료 후 자동 공개
4. 교수자가 수동으로 공개

**추가 세분화 (MVP2)**:
- 점수(숫자) 공개와 정답(내용) 공개를 별도 설정 가능
- 서술형/단답형 문항은 정답 공개 기본값 OFF

**이유**: 현재 "제출 후 정답 공개" 단일 토글은 학기 중 재응시가 많은 환경에서 정답 유출 위험이 있다. 마감 후 공개가 가장 안전하며 Canvas LMS 기본 동작과도 일치한다. 단, MVP1에서 다양한 옵션을 모두 구현하는 것은 리소스 낭비이므로 단순화한다.

---

### 2-5. 미제출 학생 처리

**결정**: "결시" 상태로 별도 처리. 0점 자동 부여 안 함.

**상세 정책**:
- 마감일 이후 미제출 학생은 `score = null`, 표시 레이블 = "미제출"
- 채점 대시보드에서 미제출 학생은 교수자가 수동으로 0점 부여 가능 (자동 부여 아님)
- 통계 계산 시 미제출 학생은 기본적으로 제외. 단, 통계 페이지에서 "미제출 포함(0점 처리)" 옵션으로 전환 가능 (MVP2)
- 학생 결과 화면에서는 "미제출" 상태로 표시

**이유**: 0점 자동 부여는 결시와 0점 실제 취득을 구분 불가하게 만든다. 별도 상태 관리가 데이터 정확성과 교수자 유연성 모두에 유리하다.

---

### 2-6. short_answer 자동채점 처리 방식

**결정**: short_answer는 수동채점 대상으로 분류. `autoGrade: 'partial'` 값은 UI 표시에 활용하되 채점 대시보드에서는 수동채점 탭으로 라우팅.

**세부 정책**:
- `autoGrade: true`: 완전 자동채점. 교수자 개입 불필요.
- `autoGrade: 'partial'`: 시스템이 정답 후보 매칭을 시도하고 결과를 교수자에게 제안. 교수자가 최종 확인/수정 후 저장. GradingDashboard에서 "AI 채점 제안" 형태로 표시 (MVP2 이후 실제 AI 연동, MVP1에서는 정확한 매칭만 자동으로 표시).
- `autoGrade: false`: 완전 수동채점.

**이유**: PM2 C-04 / PM3 H-03 에서 mockData의 'partial' 정의와 GradingDashboard의 수동처리 간 불일치가 발견되었다. 'partial'을 "제안 후 교수자 확인" 워크플로우로 명확히 정의하면 양측의 정합성이 확보된다.

---

### 2-7. 문제은행 문항 참조 vs 복사 정책

**결정**: 퀴즈에 추가 시점에 "복사본(Snapshot)" 저장 방식 채택.

**세부 정책**:
- 문제은행에서 퀴즈로 문항을 가져오면 해당 시점의 문항 내용이 퀴즈에 복사된다.
- 이후 문제은행에서 원본을 수정해도 기존 퀴즈에는 영향을 주지 않는다.
- 단, 퀴즈 편집 화면에서 "문제은행에서 새로 가져오기"를 명시적으로 선택하면 최신 버전으로 교체 가능.
- 문제은행 문항 목록에서 `usageCount`는 복사 포함된 퀴즈 수로 표시.

**이유**: 참조 방식은 문제은행 수정이 배포된 퀴즈에 소급 적용되어 학습자 형평성 문제를 야기할 수 있다. 복사 방식이 LMS 표준(Canvas 포함)에서도 일반적이며 데이터 무결성 측면에서 안전하다.

---

### 2-8. 재채점 정책

**결정**: 이미 결정된 정책 재확인 및 명문화.
- 자동채점(autoGrade=true) 문항만 재산출 대상.
- 수동으로 교수자가 입력한 점수(수동채점 문항)는 재채점 대상에서 제외, 불변 유지.
- 재채점 실행 전 영향받는 문항 목록과 학생 수를 미리 표시하는 확인 단계를 거친다.
- 재채점 완료 후 학생에게 "점수가 변경되었습니다" 알림 발송 (MVP2, 실데이터 연동 후).

---

### MVP2 범위 (다음 스프린트 — 약 4~6주 예상)

**Phase 2-A (High 우선 — 2~3주)**
- 학습자 응시 화면 (문항 유형별 응시 UI + 타이머 + 제출 플로우)
- 발행 전 체크리스트 모달
- 퀴즈 상태 자동 전이 로직 구현
- 채점 완료 → 점수 공개 → 학생 결과 열람 플로우

**Phase 2-B (Medium 우선 — 3~4주)**
- 조교 권한 설계 및 역할 기반 접근 제어
- 정답 공개 세분화 정책 구현
- 지각 제출 허용 기간/감점 설정 UI
- 코드 품질 개선 (상수 통합, React.memo, 모달 컴포넌트 중복 제거)
- 접근성 개선 (WCAG AA 기준)

---

### MVP3 후보 (향후 — 실데이터 연동 후)

- API 클라이언트 레이어 구축 및 실서버 연동
- 인증/인가 시스템 (JWT 또는 세션 기반)
- 로딩 스켈레톤 및 에러 핸들링 UI
- 리스트 가상화 (react-window)
- AI 보조 채점 기능 (short_answer 제안 채점)
- 알림 시스템 (이메일 + 인앱)
- Canvas LMS / LTI 1.3 연동
